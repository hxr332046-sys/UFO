## 3. 卡点清单（6 个，按严重度）

### 卡点 1: producePdf A0002（已知顽疾，影响最大）

**现象**:
```
[14] establish/YbbSelect/operationBusinessDataInfo [save]
    ✓ OK code=00000 rt=0  (16.8s)
    extracted:
      ybb_transition_mode: "save_then_produce_pdf"
      ybb_followup_component: "producePdf"
      ybb_followup_code: "A0002"
      ybb_followup_message: "服务端异常"
```

**根因分析**:
- YbbSelect save 本身成功（code=00000）
- 紧接着自动 follow-up 的 `producePdf` 接口返回 A0002（服务端"操作失败"通用错误）
- 之前用真实浏览器抓包，**真实浏览器也会偶发 A0002**，证明这是服务端临时异常，不是框架问题

**框架处理**:
- 已采取兜底策略：YbbSelect save 算成功，producePdf 失败被记录但不阻断 step 14
- 继续 step 15+ 流程，但 PreElectronicDoc 返回 D0018（状态已变）

**影响**:
- 服务端真实位置仍停留在 YbbSelect（producePdf 没真正完成 → 没进入 PreSubmitSuccess）
- step 17 submit 被本地 guard 阻断（这是好事）

**改进建议**:
1. 在 case 加 `producePdf_retry: true`，A0002 时自动重试 1-2 次
2. 或在 producePdf 失败时短暂 sleep 5s 再 retry
3. 服务端无 retry 友好性时，可以等用户手动操作（比框架硬撑好）

### 卡点 2: STEP_POSITION_GUARD on submit（设计行为，但应改为成功状态）

**现象**:
```
[17] establish/submit [最终提交] (optional)
    ⚠ SKIP code=STEP_POSITION_GUARD
    observed_current_comp_url: "YbbSelect"
    allowed_current_components: ["PreSubmitSuccess"]
```

**根因**:
- step 17 设计为最终提交按钮的代理调用
- 本地 guard 检查到当前服务端位置=YbbSelect 而非 PreSubmitSuccess，正确阻断

**问题**:
- 当前框架把这个状态归类为 `phase2_failed`，最终退出码 6
- 但**这正是用户要求的"到云提交为止，不真正提交"** —— 应该是成功状态

**改进建议**:
- 在 SmartRegisterRunner 加新状态 `reached_pre_submit`（已达云提交点未提交）
- exit_code 0 + status="ready_for_submit"
- 提示信息变为："✅ 已抵达云提交点，请在浏览器中点击提交"

### 卡点 3: 字号限制字检查（高频但可绕过）

**现象**:
```
[4] bannedLexiconCalibration
    extracted: tipStr="所申请的企业名称含有限制使用的内容、文字或同音字（诚）"
[5] operationBusinessDataInfo#1
    rt=2 → NameCorrectionHook 弹出交互菜单：
    [1] 更换字号  [2] 强制继续  [3] 退出

**测试历史**（哪些字号被服务端列为限制）:

| 字号 | 限制字 | 通过 |
|------|------|------|
| 鸿启智 | (无) | ✅ |
| 瑞辉达 | (无) | ✅ |

**框架处理**:
- NameCorrectionHook 弹出选项菜单

**改进建议**:
1. case 文件加 `force_continue_on_lexicon_warn: true` 配置，跳过 stdin 等待
2. 或者增加 `--auto-force` CLI 参数
3. 也可以在 case 启动前调用 `bannedLexiconCalibration` 预检字号合法性，把限制字反馈给用户

### 卡点 4: Phase 1 NameCheckInfo#3 步骤 A0002（已被 SKIP，无实际影响）

**现象**:
```
[8] NameCheckInfo/operationBusinessDataInfo#3 (optional)
    ⚠ SKIP code=A0002
    extracted: {checkState_sent: 4, continueFlag: true}

**根因**:

- 服务端识别到 force 状态，第三次提交时返回 A0002
- step 8 标记为 optional，所以 SKIP 不阻断

**影响**: 无（Phase 1 后续仍能拿到 nameId 进入 Phase 2）

### 卡点 5: SlUploadMaterial 耗时 65 秒（流程瓶颈）

**现象**: step 12（材料上传 + 绑定 + 保存）单步占用 65 秒，是总耗时 168s 的 38%。

**原因**:
- 上传 4 个文件（身份证正反面、住所证明、租赁合同）
- 每个文件需要：upload → 服务端 OCR/校验 → bind → save
- 部分文件 5MB 左右

**改进建议**:
1. 文件并发上传（当前是串行）
2. 上传前压缩文件（如果格式允许）
3. 缓存已上传文件的 contentUuid，重复 case 复用

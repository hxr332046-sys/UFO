# 登录 + 第一阶段（名称登记）技术实现总结

> 项目：广西政务平台经营主体登记自动化  
> 时间跨度：2026-04-19 → 2026-04-22  
> 作者：AI 辅助开发（Cascade）  
> 里程碑：**扫码登录 + 名称登记全流程纯 HTTP 完成**，零浏览器依赖

## 一、项目目标

在广西经营主体登记平台（9087 门户）上，自动完成从登录到名称登记的全流程。目标企业示例：

| 字段 | 值 |
|---|---|
| 企业名称 | 有为风（广西容县）软件开发中心（个人独资） |
| entType | 4540（个人独资企业） |
| 注册地 | 广西壮族自治区玉林市容县 |
| 投资人 | 黄永裕 |

平台涉及两个核心系统：
- **tyrz.zwfw.gxzf.gov.cn** — 广西统一认证（SSO 登录、QR 码）
- **zhjg.scjdglj.gxzf.gov.cn:9087** — 经营主体登记门户（业务 API）
- **zhjg.scjdglj.gxzf.gov.cn:6087** — TopIP 网关（SSO 中继）

## 二、技术演进：三代登录方案

### 第一代：浏览器 CDP + 账号密码 + 滑块（4.19 – 4.20）

```
Chrome CDP → 导航 enterprise-zone → 302 到 tyrz → 填账号密码
→ 弹出滑块 → JS DOM 模拟拖拽 → 提交 → SSO 302 链自动走完
→ 从 localStorage 读取 Authorization
```

**踩过的坑（8 个）：**

1. **CDPSession.send() 吞事件** — 封装的 WebSocket recv 循环只匹配 response.id，把 Fetch.requestPaused 等推送事件静默丢弃。解决：Fetch 阶段切换到 raw WebSocket。
2. **直接构造 tyrz URL** — 跳过 9087 直接去 tyrz 登录，9087 没有 SSO 上下文，后续 entservice 全部拒绝。教训：SSO 链不能跳步。
3. **旧 localStorage 阻止 SSO 触发** — SPA 优先读 localStorage.Authorization，有值就不触发 SSO 重定向。解决：CDP `Storage.clearDataForOrigin`。
4. **ssc.mohrss.gov.cn 阻塞** — 登录后跳到社保卡扫码页。发现 6087 SESSION 在跳 ssc 之前已设好，用 CDP Fetch 拦截 ssc 伪造 302 跳过。
5. **Fetch.disable 后残余请求卡死** — break 后子资源请求仍被 pause。解决：disable 时内联 continue 所有残余请求。
6. **CDP Input 事件对滑块无效** — 滑块库只监听 DOM MouseEvent，不响应 CDP 合成事件。解决：JS `dispatchEvent(new MouseEvent(...))`。
7. **滑块图片面板默认隐藏** — `display:none` 导致 naturalWidth=0。解决：JS 强制设 `display:block`。
8. **滑块连续失败后图片返回空** — 服务端限流。解决：指数退避重试。

**致命缺点**：需要 Chrome 持续运行，CDP 连接脆弱，滑块识别不稳定。

### 第二代：纯 HTTP 扫码 + CDP 收尾（4.21 – 4.22 白天）

```
Python requests:
  Step 1-5: tyrz QR API → 获取二维码 → 手机扫码 → 提交登录 → SESSIONFORTYRZ
浏览器 CDP:
  Step 6: 注入 SESSIONFORTYRZ cookie → 导航 enterprise-zone → SSO 自动完成 → 读 Authorization
```

**踩过的坑（6 个）：**

1. **tyrz `/sso/oauth2/authorize` 返回 nginx 404** — 这是历时最长的问题。submitLogin 返回的 redirect URL 走 `/sso/` 路径，从外部任何方式访问都是 404。根因：`/sso/` 只在服务端内部 upstream 间可达，对外只暴露 `/am/`。当时误以为是 header/参数问题，排查了 DNS、代理、Accept 头等方向，全部无效。最终结论：放弃 `/sso/` 路径。
2. **HTTP 请求 tyrz 超时卡死** — 系统代理未关。解决：`s.proxies = {"https": None}`。
3. **6087 authLogin 返回 JSON 而非 302** — requests 默认 `Accept: application/json`，服务器据此返回 JSON 错误体。改 `Accept: text/html` 后正常 302。
4. **Fetch 事件堆积导致浏览器挂起** — sleep 期间 requestPaused 无人处理。解决：主动消息循环替代 sleep。
5. **重复登录触发风控** — 短时间内多次扫码，QR 接口响应变慢。教训：登录只做一次。
6. **checkQrCode 参数名** — 不是 `sessionId` 而是 `random`，必须逆向 JS 确认。

**改进**：扫码稳定，不受滑块影响。但仍依赖 Chrome 做最后一步。

### 第三代：完全纯 HTTP（4.22 深夜 ★最终方案）

```
Python requests 全程:
  Step 1: QR 扫码 → SESSIONFORTYRZ
  Step 2: entservice → 6087 bridge → tyrz /am/ 授权 → 6087 SESSION
  Step 3: 清 icpsp-api cookie → entservice → sign bridge → ENT_SERVICE session
  Step 4: entservice → sign bridge → redirectUrl 含 Authorization ✓
```

**三个关键突破：**

#### 突破 1：`/am/oauth2/authorize` 可从外部访问

之前被 `/sso/oauth2/authorize`（404）误导，以为纯 HTTP 无法走完 SSO 链。关键发现：正常 SSO 入口走 `/am/`，不是 `/sso/`。带 SESSIONFORTYRZ 请求 `/am/` 路径，tyrz 自动授权返回 code。

#### 突破 2：Bridge "加载中" 页面的逆向

6087 和 9087 的 SSO 中间页都是同一个 bridge 模式：
- **GET** → 返回 "加载中" HTML（含 `tokenKey` 和 `xhr.open`）
- **POST**（带 `X-Requested-With: XMLHttpRequest`）→ 返回 JSON

JSON 响应结构：
```json
{"code": "0",  "data": {"redirectUrl": "http://...?Authorization=xxx"}}  // 成功
{"code": "-9", "data": {"redirect_uri": "http://..."}}                   // 需要 redirect 授权
```

Python 模拟 bridge JS 行为：检测到 bridge HTML → POST 同一 URL 带 XHR header → 解析 JSON。

#### 突破 3：两轮 entservice?sign

**这是整个项目最关键的发现。**

entservice 的两阶段设计：
1. **第一次 sign** → 建立 9087 session（`lastAuthType=ENT_SERVICE`），redirectUrl 只到 `/TopIP/`，无 Authorization
2. **第二次 sign** → 检测到已有 ENT_SERVICE session → 生成 Authorization 放入 redirectUrl

```
entservice?sign 第一轮:
  GET → bridge HTML → POST → {"code":"0", "data":{"redirectUrl":"/TopIP/"}}
  
entservice?sign 第二轮:
  GET → bridge HTML → POST → {"code":"0", "data":{"redirectUrl":"portal.html#/login/authPage?Authorization=b00a0a4d..."}}
```

**成果**：从此整个登录流程零浏览器依赖。

## 三、登录态三层架构

```
┌──────────────────────────────────────────────┐
│  ① SESSIONFORTYRZ (tyrz 域)                  │
│  · 扫码建立                                    │
│  · 有效期：数小时                               │
│  · 过期 → 必须重新扫码                          │
├──────────────────────────────────────────────┤
│  ② 6087 SESSION (TopIP 网关)                  │
│  · SSO 链建立                                  │
│  · 有效期：~2h                                 │
│  · 过期 → 用①自动重建                          │
├──────────────────────────────────────────────┤
│  ③ Authorization (32位hex, 9087 API 用)       │
│  · entservice 两轮 sign 生成                    │
│  · 有效期：~30min – 2h                         │
│  · 过期 → 用①②自动重建                         │
│  · 存储：runtime_auth_headers.json             │
└──────────────────────────────────────────────┘
```

三级自动恢复策略（`ensure_token()`）：
```
token 还活？ → 直接用（0 秒）
SESSIONFORTYRZ 还在？ → 静默续期（~2 秒，无需扫码）
全部过期？ → QR 扫码（~30 秒，需手机）
```

## 四、第一阶段：名称登记（纯协议驱动器）

### 架构思想：Browser Distillation（浏览器蒸馏）

```
浏览器的唯一作用 = 无（连登录都不需要了）
全部逻辑 = Python requests + Authorization header
```

### 7 步 API 调用链

| 步骤 | API | 作用 | 关键发现 |
|---|---|---|---|
| 1 | `checkEstablishName` | 检查企业类型是否可设立 | guide 预判，可选 |
| 2 | `loadCurrentLocationInfo` | 初始化流程定位 | body 必须精简，多余字段导致下游 D0022 |
| 3 | `loadBusinessDataInfo` | 加载 NameCheckInfo 组件 | 必须含 `itemId: ""`（空字符串） |
| 4 | `bannedLexiconCalibration` | 禁限用词检查 | 辅助查询，失败不致命 |
| 5 | `operationBusinessDataInfo#first` | 首次保存名称信息 | resultType=2 表示需要查重 |
| 6 | `nameCheckRepeat` | 名称库查重 | 返回近似名列表 + checkState |
| 7 | `operationBusinessDataInfo#second` | 二次保存（含查重结果） | **必须带 `afterNameCheckSign: "Y"` + `nameCheckDTO`**，否则不分配 busiId |

### Phase 1 踩坑记录

#### 坑 1：D0022 越权访问 — 三层洋葱

D0022 是一个**万能垃圾桶错误码**，header 不对、body 不对、signInfo 不对都返回同一个 D0022，没有任何更具体的提示。排查如同剥洋葱：

| 层 | 根因 | 排查时间 |
|---|---|---|
| 第一层 | 缺浏览器指纹头 (`sec-ch-ua` 等 6 个 header) | 2h |
| 第二层 | 请求 body 多/少字段（`loadCurrentLocationInfo` 多了 `extraDto`；`loadBusinessDataInfo` 漏了 `itemId`） | 4h |
| 第三层 | `signInfo` 魔数不对（必须是 `-252238669`） | 3h |

方法论 — "样本即真理"四步法：
1. 用 mitm 抓浏览器真实请求作为**真理源**
2. 与 Python 构造的请求做**递归字段级离线 diff**（不发请求）
3. 逐字段修正，一次改一个
4. 绝不猜测，diff 能解决 90% 的问题

#### 坑 2：signInfo 是固定魔数

最初以为 signInfo 是根据请求 body 动态计算的 Java hashCode 签名。实现了 Python 版 Java hashCode 算法后，动态算出的值仍然被 D0022 拒绝。

真相：mitm 备份 4 个首次保存样本的 signInfo **全部是同一个值 `-252238669`**。它不依赖 body 内容、不依赖用户，是服务端内置的"已知合法魔数"。直接硬编码即可。

#### 坑 3：第二次保存必须带 afterNameCheckSign 和 nameCheckDTO

第 7 步（二次保存）如果只发 name 字段，服务端返回 `resultType=2` 但不分配 busiId。必须同时带：
- `afterNameCheckSign: "Y"` — 表示"用户已确认近似名"
- `nameCheckDTO` — 第 6 步 nameCheckRepeat 返回的完整 busiData

这两个字段的要求在前端 JS 中有，但 API 文档完全没提。纯靠 mitm 样本对比发现。

#### 坑 4：loadCurrentLocationInfo body 必须精简

浏览器实际发送的 body 只有 `{flowData: {busiId, busiType, entType}, linkData: {token: ""}}`。
driver 最初构造了更多字段（`extraDto`, `vipChannel`, `continueFlag` 等），导致服务端 SESSION 被推到错误分支，**下游所有请求都 D0022**。

教训：多字段不等于安全，精简到样本一致才是正确做法。

#### 坑 5：D0029 操作频繁

连续调试发送过多请求触发服务端限流。D0029 后必须等 5-10 分钟冷却。driver 内置步间 0.9s 节流 + D0029 熔断自动停止。

## 五、错误码词典

| 错误码 | 含义 | 应对 |
|---|---|---|
| `00000` | 成功 | - |
| `GS52010103E0302` | 未认证/会话无效 | 重新获取 Authorization |
| `D0022` | 越权访问（万能拒绝码） | 检查 header → body → signInfo |
| `D0029` | 操作频繁（服务端限流） | 等 5-10 分钟，不要重试 |
| `A0002` | 业务数据异常 | 检查请求体字段格式（如 busiAreaData 编码问题） |

## 六、关键方法论

### 方法论 1：样本即真理

面对"服务端黑盒 + 万能错误码"场景，唯一可靠方法是用浏览器真实请求作为真理源，通过离线 diff 找差异，逐字段修正。绝不猜测。

### 方法论 2：单步推进

实网调试纪律：执行一步 → 等 2-5 秒 → 读反馈 → 确认成功后再下一步。失败立即停止分析。每一次请求都有代价。

### 方法论 3：Browser Distillation（浏览器蒸馏）

把浏览器的角色从"操作载体"降级为"抓包参考"，最终连登录都不需要浏览器。全部业务逻辑用 Python requests 实现，速度从分钟级降到秒级。

### 方法论 4：逆向 JS 是刚需

参数名不能猜（`random` vs `sessionId`）、bridge 页面行为不能猜（POST 同一 URL）、signInfo 来源不能猜。**每一个不确定的字段都必须回溯到 JS 源码或 mitm 样本**。

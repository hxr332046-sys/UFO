## 1. 立项背景

v2.1 之前的框架是 "case 写什么传什么，服务端拒了才知道" 的盲跑模式：

| 问题 | 表现 |
|------|------|
| 看不见枚举边界 | 政治面貌写 "99" → 跑到 PersonInfoRegGT 才报错 |
| 行业代码失配 | industry_code=6513（软件）但描述 "餐饮" → 服务端二次审核拒绝 |
| 必填字段遗漏 | person.cerNo 为空 → 跑到 MemberBaseInfo 才发现 |
| 服务端突变 | 新平台返回未知 code → 框架闷头继续，最终在某步骤异常崩溃 |

每次失败都要回溯排查 → 浪费时间 + 服务端积累垃圾数据。

---

## 2. 设计目标

> "case JSON 写错 → 启动前就停；服务端突发新错 → 运行中就停；常用合法值 → 跑一次自动入库"

| 目标 | 实现机制 |
|------|---------|
| 全覆盖 | OptionDict 30 字段 820+ 项 + IndustryMatcher 1971 行业 |
| 智能纠错 | code/name/keyword 三态匹配，候选 top-3 输出 |
| 不确定停下 | UncertaintyHook 监控未知 code/关键词/rt=2 |
| 持续学习 | OptionsScout 每次 load 被动建库 |
| 多重保障 | 国标包 + 服务端实测 + Scout 反推 三重数据源 |

## 4. 核心模块详解

### 4.1 OptionDict — 字典中枢

**文件**: `system/governance/option_dict.py`

**职责**:
- 加载 / 保存 `data/options_dict.json`
- 单例懒加载（线程安全）
- 字段查询、code/name 三态校验、模糊关键词匹配
- 反哺接口（Scout / 标准包导入器都用）

**核心 API**:

```python
from governance import OptionDict

# 加载
od = OptionDict.load()

# 查询
f = od.get_field("politicalStatus")
opt = f.find_by_code("01")           # → OptionEntry(code='01', name='中共党员')
opt = f.find_by_name("中共党员")      # → 反向命中
opts = f.search_by_keyword("党员")    # → 模糊搜索

# 三态校验
status, matched, candidates = od.validate_value("cerType", "10")
# status: 'pass' | 'unknown_field' | 'open_field' | 'fail' | 'ambiguous'

# 反哺（Scout 自动调用）
od.upsert_options("politicalStatus",
                  options=[{"code": "14", "name": "新增项"}],
                  source="MyComponent.busiData")
od.save()
```

**数据结构**:

```python
@dataclass
class OptionEntry:
    code: str
    name: str
    extra: Dict[str, Any] = {}      # parent / depth / ent_type / etc.

@dataclass
class FieldOptions:
    field_name: str
    label: str
    source: str                     # 来源标识，便于追溯
    options: List[OptionEntry]
    is_open: bool = False           # True = 自由文本字段（如 entName）
    last_seen_iso: str = ""
```

**校验状态返回值**:

| status | 含义 | 何时出现 |
|--------|------|---------|
| `pass` | 命中合法选项 | code/name 精确匹配 |
| `unknown_field` | 字典未收录该字段 | 该字段不在 fields 字典里 |
| `open_field` | 自由文本字段 | is_open=True |
| `fail` | 完全不命中 | 无任何匹配 |
| `ambiguous` | 多候选 | 关键词命中 ≥ 2 项 |

### 4.2 IndustryMatcher — 行业模糊匹配引擎

**文件**: `system/governance/industry_matcher.py`
**数据**: `data/industry_gb4754_2017.json` (1971 节点 / 1381 个 4 位小类)

**三层匹配策略**:

| 层 | 策略 | 示例 | score |
|---|------|------|-------|
| 1 | 精确 name 匹配 | "应用软件开发" → 6513 | 1.0 |
| 2 | 关键词包含 | "软件开发" → 6511/6513 等 | 0.65~0.95 |
| 3 | 同义词扩展 | "餐饮" → 6110/6241/6299 | 0.65~0.85 |
| 4 | 字符+子序列 fallback | 任何残缺输入 | 0.4~0.6 |

**内置同义词组**:
```python
SYNONYM_GROUPS = {
    "软件":   ["软件", "信息技术", "IT", "互联网"],
    "商贸":   ["商贸", "贸易", "商品", "批发", "零售"],
    "餐饮":   ["餐饮", "饭店", "食品", "烹饪"],
    "咨询":   ["咨询", "顾问", "服务"],
    "建筑":   ["建筑", "建造", "施工"],
    "运输":   ["运输", "物流", "货运"],
    "教育":   ["教育", "培训", "学习"],
}
```

**API 用法**:

```python
from governance import IndustryMatcher

m = IndustryMatcher.load()

# 1. 精确查代码
node = m.lookup_by_code("6513")
# IndustryNode(code='6513', name='应用软件开发', parent='651', depth=4)

# 2. 校验代码合法性
m.is_valid_code("6513")    # True
m.is_valid_code("9999")    # False（不在 1971 节点里）

# 3. 文本搜索 top-N
results = m.search("软件开发", top_n=3)
# [
#   IndustryMatch(code='6511', name='基础软件开发', score=0.85, reason='keyword'),
#   IndustryMatch(code='6513', name='应用软件开发', score=0.85, reason='keyword'),
#   IndustryMatch(code='6512', name='支撑软件开发', score=0.85, reason='keyword'),
# ]

# 4. 决断性判断
m.is_decisive(results)
# False — 三个 score 一致 → 需要询问用户
```

### 4.3 CaseValidator — 启动前质检

**文件**: `system/governance/case_validator.py`

**4 维校验**:

1. **必填检查**（含别名）
2. **枚举字段校验**（OptionDict）
3. **行业代码校验**（GB/T 4754 树内）
4. **行业语义一致性**（描述 vs 代码）

**必填字段定义**（带别名）:

```python
REQUIRED_FIELDS = [
    ("case_id", []),
    ("company_name_full", []),
    ("name_mark", []),
    ("phase1_check_name", []),
    ("entType_default", []),
    ("phase1_industry_code", ["itemIndustryTypeCode"]),
    ("phase1_main_business_desc", ["business_area", "phase1_industry_special"]),
    ("phase1_dist_codes", []),
    ("address_full", ["dom", "domAddress"]),
    ("person.id_no", ["person.cerNo", "person.idNo", "person.idCard"]),
    ("person.name", ["person.memName", "person.fullName"]),
]
```

**枚举字段映射**（57 条别名规则，详见 §7）:

```python
CASE_FIELD_TO_OPTION_KEY = {
    "entType_default": "entType",
    "person.politicsVisage": "politicalStatus",
    "person.cerType": "cerType",
    # ... 共 30+ 条
}
```

**API 用法**:

```python
from governance import CaseValidator
v = CaseValidator()
report = v.validate(case)

# 报告字段
report.checked_fields    # 检查总数
report.fails             # FAIL 列表
report.ambiguous         # AMBIGUOUS 列表
report.warns             # WARN 列表
report.is_blocking()     # 是否需要阻断

# 交互式修复 ambiguous
new_case = v.interactive_resolve(report, case)
```

**ValidationIssue 结构**:

```python
@dataclass
class ValidationIssue:
    field_path: str          # case 字段路径，如 "person.cerNo"
    case_value: Any          # 原值
    level: IssueLevel        # pass/ambiguous/fail/warn
    kind: IssueKind          # missing_required/invalid_enum/...
    message: str             # 人类可读描述
    candidates: List[Dict]   # 候选合法值
    suggestion: Dict         # 最佳建议
    auto_fixable: bool       # 可自动修复
```

### 4.4 OptionsScout — 被动反向扫描器

**文件**: `system/governance/options_scout.py`

**5 类识别策略**:

| 策略 | 触发条件 | 示例 |
|------|---------|------|
| 1. fieldList 元数据 | `fieldList: [{...field, fieldDesc, mustFlag}]` | BasicInfo 45 字段 |
| 2. List 后缀枚举 | 字段名以 List/Options/Arr/Enum/Items 结尾 + 数组+code+name | `politicalStatusList` |
| 3. 隐藏枚举数组 | 数组元素含 code+name 双键且 ≥ 2 项（无 List 后缀） | `uploadMaterialVo` |
| 4. 角色字典 | 所有 key 形如大写字母数字 (FR05/CWFZR) | `pkAndMem` |
| 5. xxxCode + xxxCodeName 配对 | 同层级两个字段 | `nationalityCode=156` + `nationalityCodeName=中国` |

**核心字段名识别**:

```python
CODE_KEYS = ("code", "value", "key", "id", "itemValue", "dictCode",

NAME_KEYS = ("name", "label", "text", "title", "displayName",
             "itemText", "dictName", "optionName", "tabName", "stateName")
```

```python
# system/icpsp_api_client.py
class ICPSPClient:

    def register_response_observer(self, ob: Callable[[str, dict, dict], None]):
        self._response_observers.append(ob)
```

### 4.5 UncertaintyHook — 运行时停下钩子

**文件**: `system/governance/uncertainty_hook.py`

**三类触发条件**:

```python
class UncertaintyHook:
    def __init__(self):
        self.priority = 45      # 在 ThrottleHook(40) 之后，StateExtractor(50) 之前

    KEYWORDS = ["请选择", "不在范围", "不存在", "无效", "必填",
                "格式错误", "应为", "不允许"]

    BENIGN_CODES = ("00000", "D0010", "D0018", "D0021", "SKIP")

    def post_response(self, ctx, step, resp):
        code = resp.get("code", "")

        # 1. 未知 code（不在 ERROR_DICT，不在 BENIGN_CODES）
        if code not in BENIGN_CODES and code not in known_error_dict:
            raise InterventionSignal(kind="uncertain_unknown_code", ...)

        # 2. 关键词命中
        msg = resp.get("msg", "") + str(...)
        for kw in KEYWORDS:
            if kw in msg:
                raise InterventionSignal(kind="uncertain_keyword", ...)

        # 3. rt=2 出现在非核名步
        if resp.get("data", {}).get("resultType") == "2":
            if step.tag not in ("p1_core",):
                raise InterventionSignal(kind="uncertain_rt2_outside_namecheck", ...)
```

**InterventionSignal**:

```python
class InterventionSignal(Exception):
    kind: str                       # uncertain_*
    options: List[str] = ["abort", "continue", "retry"]
    message: str
    context: Dict[str, Any]
```

Pipeline 接住信号 → 上报 Runner → Runner 决定是 abort/continue/retry。

## 6. SmartRegisterRunner 集成

### 6.1 启用治理（默认）

```python
runner = SmartRegisterRunner(
    case_path=Path("docs/case_xxx.json"),
    enable_governance=True,        # 默认 True
    strict_validation=False,       # True: ambiguous 也阻断
)
runner.execute()
```

### 6.2 执行顺序（治理介入点）

```
0.   load_case
0a.  ★ _run_case_validation        ← CaseValidator 启动前质检
       ├─ pass → 继续
       ├─ fail → 终止（exit 2）
       └─ ambiguous + strict → 交互修复
0.0  qr_login / silent_session_refresh
0.1  check_auth
       └─ 创建 client → ★ _attach_governance_scout
0.2  check_duplicate
1.   run_phase1                    （Scout 被动收集）
2.   run_name_completion
3.   run_phase2                    （Scout 被动收集）
4.   ★ _persist_governance_scout   ← 写回字典
```

### 6.3 关闭治理（fallback v2.1 行为）

```python
runner = SmartRegisterRunner(
    case_path=...,
    enable_governance=False,
)
```

## 12. FAQ

### Q1. 字典是原项目的吗？还是新做的？

**两者都有**：
- 原项目已有：phase1_service 下 6 类字典 + schemas/ + data/industry_gb4754_2017.json
- 新合并：v2.2.5 把它们全部整合到 options_dict.json
- 新增：国标包（national_standard_enums.json）+ Scout 反推数据

### Q2. 服务端有字典 API 吗？

**没有发现**。探针测试 252 个组合都失败（详见 `scripts/_probe_dictionary_apis.py` 注释）。
服务端的字典数据通过：
- 业务接口（loadBusinessDataInfo）随响应返回
- xxxCode + xxxCodeName 配对内嵌
- 前端 SPA chunk 硬编码（提取困难，大部分集中在 i18n key-value 而非紧凑对象）

### Q3. 跑一次实际流程会发现多少新字段？

**实测**：13 个新字段（v2.2.2 真实跑通后）。每跑不同的 case 或 entType 还能继续发现，是**持续学习**机制。

### Q4. 国标包的值跟服务端实测冲突怎么办？

**实测优先**：
- OptionDict.upsert_options 默认不覆盖已有 code
- 国标包导入器是 upsert 模式
- Scout 实测的 code 永远保留
- 国标包只补充缺失的 code

### Q5. ambiguous 怎么处理？

两种模式：

```python
# 1. 严格模式（生产用）
runner = SmartRegisterRunner(
    case_path=...,
    strict_validation=True,    # ambiguous 也阻断
)

# 2. 宽松模式（默认，开发用）
runner = SmartRegisterRunner(
    case_path=...,
    strict_validation=False,   # 仅 fail 阻断
)
```

严格模式会调用 `v.interactive_resolve(report, case)` 让用户从候选里选。

### Q6. 新省份/新业务类型怎么扩展？

1. 改 case 的 `entType_default` / `phase1_dist_codes` / 等
2. 跑一次完整流程 → Scout 自动建库
3. 看 `data/options_dict.json` 是否需要补字段（手动 upsert）
4. 看 IndustryMatcher 是否需要补同义词

### Q7. 字典加载性能如何？

- OptionDict 单例懒加载，整个进程只加载一次
- 加载 820+ 项约 50ms
- IndustryMatcher 加载 1971 节点约 200ms
- Scout 每次 load 增量处理（响应到达即识别）

### Q8. 治理子系统会让流程变慢吗？

**几乎不会**。
- 启动前质检：~100ms（一次性）
- 运行中 Scout：每个 load 响应增加 < 5ms
- UncertaintyHook：每次响应增加 < 1ms

### Q9. 如何离线扫历史抓包？

```bash
# 把抓包响应放入 packet_lab/out/component_loads/
# 然后跑：
.\.venv-portal\Scripts\python.exe scripts\_real_scout_run.py
```

### Q10. 治理子系统能完全关闭吗？

可以，三种方式：
1. `enable_governance=False`
2. 删除 `data/options_dict.json` (会重建空字典)
3. 不调用 `register_response_observer`

完全回退到 v2.1 行为。

## 三、缺口分析（按优先级）

### P0 关键缺口（LLM 无法完成完整注册）

1. **NameSupplement 协议链（截图12）**
   - 信息补充 = Phase1 的 Step2，必须提交才能到"申报完成"
   - 需要字段：经营范围选项、出资额、拟登记机关、投资人信息、经办人信息
   - 技术方案：新增 `POST /api/phase1/supplement` 端点
   - 依赖：busiId（来自 register 成功返回）

2. **申报完成（Step3）**
   - 用户截图左侧导航 1→2→3 进度条的最后一步
   - 提交后才真正完成"名称登记"阶段
   - 技术方案：新增 `POST /api/phase1/submit_name` 端点

### P1 功能缺口（影响覆盖面但不阻塞核心流程）

3. **"已办理名称预保留"路径（截图3）**
   - 当前只走"未申请"，不支持已有保留号的企业
   - 需要：`POST /api/phase1/register_with_reservation`

4. **知名商标禁用词补充（截图7-8）**
   - "美的"（驰名商标）未在本地词库
   - 需要：从 bannedLexiconCalibration API 批量抓取常见商标词
   - 或在 banned_words.json 补充 top 100 驰名商标

5. **经营范围审批类型标签（截图11）**
   - 红色=前置审批（设立前需许可）
   - 黄色=后置审批（成立后需备案）
   - 当前 scope 返回的 raw 数据可能已包含此字段，需解析

6. **分支机构特殊字段**
   - fzjg 需要 parentEntRegno（隶属企业统一信用代码）
   - Phase1Case schema 需暴露此字段

### P2 体验优化（LLM友好度）

7. **统一 CLI 入口**
   - 当前是 HTTP API，LLM 需通过 function calling
   - 可加 `python -m phase1_service.cli register --case case.json`

8. **建议推荐系统**
   - 字号被拒后，自动推荐相似可用字号
   - 基于 similar_names 的冲突分析

## 四、回答你的核心问题

### Q1: "所有环节所有细节都普查记录在案了吗？"

**Step 1 名称检查 — 95% 完成**：
- ✅ 11 种企业类型 + 10 种 industries/organizes 字典
- ✅ 7 步协议链 → busiId
- ✅ 禁限用词预检（444 词本地 + 远程校验）

**Step 3 申报完成 — 0% 完成**

### Q2: "通过 API 都能注册登记成功是不是？"

**当前**：能完成到 **拿到 busiId**（名称检查通过），但不能完成完整的"名称登记"。
缺最后两步：信息补充 + 申报完成。

### Q3: "不成功会不会返回错误和推荐？"

**是的**，已实现 12 种结构化错误分类：
- `name_prohibited` — 含禁止词（附命中词列表）
- `name_restricted` — 含限制词（附命中词+分类）
- `name_conflict` — 同名冲突过多
- `rate_limited` — 服务端限流（附冷却建议）
- `auth_expired` — 认证过期（提示重新登录）
- `privilege_denied` — 越权访问
- 等等...

每个错误都有 `error_code` + `reason` + `reason_detail`，LLM 可直接解析并给用户建议。

### Q4: "做成 CLI 给 LLM 使用？"

架构完全支持。当前 API 已经是：
- **输入标准化**：Phase1Case schema 定义了所有必填/选填字段
- **输出结构化**：JSON response 含 success/error_code/reason/similar_names
- **幂等安全**：同 case 30分钟内不会重复创建草稿
- **字典自足**：所有下拉选项都有离线字典，LLM 不需要看 UI

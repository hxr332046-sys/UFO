# 加固路线图：要做的事

> **性质**：本文件列出为提升政务自动化「坚固性」**应完成的工作项**，与 `LLM政务自动化框架_目标与实施标准.md`、`逆向复现方法论_上下文先行与数据填充.md` 配套执行。  
> **维护**：完成一项可在下文勾选或改日期；重大阶段完成后更新 `LLM政务自动化框架` 中「当前阶段」说明。

---

## 目标：什么叫「坚固」

| 级别 | 含义 | 是否已达成（自查） |
|------|------|-------------------|
| **可用** | 指定环境下、按文档操作能跑通到约定停点 | ☐ |
| **可复用** | 他人机器按同一文档可复现；失败有可读 JSON/记录 | ☐ |
| **可演进** | 站点小改时主要改「状态/断言映射」，而非全盘重写脚本 | ☐ |
| **可宣称** | 对外说「支持某业务范围」时有 **普查矩阵 + 回归用例** 背书 | ☐ |

---

## 一、环境钉死（优先；否则后续全是噪声）

| 序号 | 要做的事 | 说明 |
|------|-----------|------|
| E1 | 固定 **HTTPS 可访问** | 系统时间、证书；必要时使用「忽略证书错误」启动方式；消除 `ERR_CERT_*` 长期阻塞。 |
| E2 | 固定 **登录与会话** | 单实例 Chrome / 少标签；约定「已登录办件中心」后再跑自动化；文档写明操作顺序。 |
| E3 | **Token/401 策略** | CDP 读 `localStorage` 或与网关协同；401 时重试路径写清，避免静默失效。 |
| E4 | **网络与风控** | 区分业务错误与出口 IP/风控；后者用 `census_run_record` 记现象，不当作脚本逻辑 bug 无限改。 |

**验收建议**：同一台机器连续三次冷启动，能稳定进入同一 **`l3` 起点**（如已登录门户或 `--resume-current` 的 core）。

---

## 二、执行层：状态机 + 前置条件（从「乱点主按钮」升级）

| 序号 | 要做的事 | 说明 |
|------|-----------|------|
| S1 | 画 **设立主路径状态图** | 门户 → 专区 / name-register → `guide/base`（若出现）→ `core` 各 hash → 云提交**文案停点**；状态命名稳定。 |
| S2 | 每步定义 **前置条件** | 例：未关 MessageBox 不点下一步；级联未完成不提交。 |
| S3 | **主循环对齐状态机** | `packet_chain` 等由「纯贪心主按钮」逐步改为 **按当前状态选白名单动作**。 |
| S4 | 卡住时可解释 | 输出必须能回答：**当前状态名 + 缺哪条前置**，而非仅「没点着」。 |

---

## 三、断言与回归（把「以为成功」变成「可证明成功」）

| 序号 | 要做的事 | 说明 |
|------|-----------|------|
| A1 | **接口断言模板** | 关键 `icpsp-api`：`code` / 必要业务字段；与 `replay_lab` / `mitm_replay_core` 衔接。 |
| A2 | **页面断言摘要** | `hash` + 必要 DOM/文案与步骤绑定；与 `blocker_evidence` / `snap` 一致。 |
| A3 | **小型回归集** | 3～5 条固定路径；脚本或 CI 可跑；**改代码必跑**，防回退。 |

---

## 四、普查矩阵收口（证据驱动「可宣称」）

| 序号 | 要做的事 | 说明 |
|------|-----------|------|
| C1 | 维护 **`busiType × entType × 入口`** 必测格表 | 每格至少一次真实跑 + `census_run_record`（或等价 JSON）。 |
| C2 | **未测格显式标红** | 未闭合不写入「已支持」对外表述。 |
| C3 | 与 **字典 V2 / 死角清单** 对账 | 新发现并入 §6/§7 类文档。 |

---

## 五、数据与材料（与步骤绑定，而非只绑「资料 txt」）

| 序号 | 要做的事 | 说明 |
|------|-----------|------|
| D1 | **case JSON / material_pack** 映射到「步骤 + 控件槽位」 | 换案例只换数据文件，少改 Python。 |
| D2 | **复杂控件专项** | 级联、树选、iframe 等单独小脚本 + 记录，不混进唯一大杂烩。 |
| D3 | 合规规则与政务校验反馈 | 逐步从「关键词」升级到可解释规则（长期）。 |

---

## 六、LLM（殿后；骨架坚固后再接）

| 序号 | 要做的事 | 说明 |
|------|-----------|------ |
| L1 | **Observation + digest + 状态** 作唯一规划输入 | 输出结构化 **Plan**，程序校验后再调工具。 |
| L2 | **禁止** LLM 无校验直连 CDP | 与 HITL、确定性执行原则一致。 |
| L3 | 验收：**关掉 LLM** 仍能靠脚本到约定停点 | 智能是增量，不是底座。 |

---

## 七、推荐实施顺序（最短路径）

1. **环境（一）** → 2. **状态机 + 前置（二）** → 3. **断言与回归（三）**  
4. **普查矩阵（四）** → 5. **数据绑定（五）** → 6. **LLM（六）**

与「先普查、再 LLM」「上下文先行、框架撑骨、数据后填」一致。

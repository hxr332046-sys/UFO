# 专业逆向与自动化流程（SOP）

> **地位**：本仓库**唯一推荐的工作流程入口**。其它文档（框架标准、纪律、两阶段业务）是细则；**先按本 SOP 分阶段推进，再落到具体脚本**。  
> **目标**：把「乱搞」变成**可重复、可验收、可复盘**的专业闭环：环境 → 取证 → 建模 → 自动化边界 → 证据与回归。

---

## 0. 总原则（不可省略）

| 原则 | 含义 |
|------|------|
| **先观测，后自动化** | 没有 API/路由/状态清单就写主链，一律视为不合格。 |
| **数据包与 CDP 分工** | **接口契约、错误码、顺序**靠抓包/重放搞清楚；**上传、复杂 SPA、实人闸门前**靠 CDP。二者都写进证据，不互相替代。 |
| **两阶段业务不混** | 名称登记（第一阶段）与「我的办件 → 继续办理」后的长表单（第二阶段）须**分别验收**，见 [设立登记两阶段流程.md](./设立登记两阶段流程.md)。 |
| **类人节奏与串行** | 实网：**单步 → 延时 → 读反馈 → 下一步**；禁止无上限循环、禁止并发副作用；见 [实网开发操作纪律_单步与风控.md](./实网开发操作纪律_单步与风控.md)。 |
| **凭证与合规** | Token/密码不进 git；自动化停在**云提交文案**前，不代点云处理（与框架标准一致）。 |

## 1. 阶段划分与门禁（建议顺序）

### 阶段 A — 环境与可复现入口（门禁：否则后面全是噪声）

- **内容**：本机时间、HTTPS、证书策略；Chrome Dev **单实例**、`config/browser.json` 与 CDP 端口一致；能稳定打开 9087 业务页（非 `chrome-error://`）。  
- **产出**：在验证记录或 `census_run_record` 里写一句「环境 OK + 日期」。  
- **参考**：[类人边界与实战约定.md](./类人边界与实战约定.md)、`config/browser.json`。

**门禁**：未过 A，**禁止**对 9087 跑长脚本或压 `--human-fast`。

---

### 阶段 B — 被动取证（数据包优先）

- **内容**：在**人工或单次 CDP** 走通目标路径时，用 **DevTools Network / mitm** 导出关键请求列表：URL、method、关键 header 名（不写全量密钥）、典型 `code` 与错误体。  
- **产出**：`packet_lab` / `survey` 下可检索的**接口目录或笔记**（或沿用现有 `auth_analysis` 类产出）；标注「与哪个 `hash` / 步骤对应」。  
- **参考**：框架 §观测层、[逆向复现方法论_上下文先行与数据填充.md](./逆向复现方法论_上下文先行与数据填充.md)。

**门禁**：目标路径上**主接口清单为空**时，不进入「写自动化主链」阶段。

---

### 阶段 C — 状态机与业务验收（建模）

- **内容**：画**有限状态**（门户 → 名称/办件 → `guide/base`? → `core` → 云提交停点）；每步写**前置条件**（已登录、办件 ID、名称核准结果等）。  
- **产出**：短文或表（可放在 `docs/` 或 `survey/`），并与 [加固路线图_要做的事.md](./加固路线图_要做的事.md) 状态机条目对齐。  
- **业务特例**：第一阶段是否成功，以**我的办件列表是否出现资料中的企业名称**等为验收，见 [设立登记两阶段流程.md](./设立登记两阶段流程.md)。

**门禁**：状态机里若「第一阶段 / 第二阶段」仍混成一条线，**禁止**宣称「设立流程已支持」。

---

### 阶段 D — 自动化实现边界（写代码）

- **CDP**：只做 B/C 证明**必须浏览器**的部分；动作间 `sleep_human`，入口 `configure_human_pacing`。  
- **接口**：重放/断言放在 `replay_lab`、客户端封装等，**与页面步骤绑定**（便于回归）。  
- **禁止**：`while True` 实网试到成功、多进程同时打 9087、实网长期 `--human-fast`。  
- **参考**：`.cursor/rules/gov-cdp-pacing.mdc`、`system/human_pacing.py`。

**门禁**：新脚本必须在 README 或 `docs/` 中**一句话说明**属于 D 中哪条路径（纯 CDP / 纯重放 / 混合）。

---

### 阶段 E — 证据与普查矩阵（验收）

- **内容**：每次真实跑复制 `census_run_record` 模板，填写 `observed_issues[]`；重要跑批保留 `case_run_*_latest.json` 或等价物。  
- **产出**：**可指向路径的 JSON** + 必要时截图目录说明。  
- **参考**：[LLM政务自动化框架_目标与实施标准.md](./LLM政务自动化框架_目标与实施标准.md) §普查、`dashboard/data/records/census_run_record.v1.example.json`。

**门禁**：无记录的真实跑，**不得**作为「已支持某业务」的依据。

---

### 阶段 F — LLM 与产品化（仅在后置条件满足后）

- **内容**：仅在 A–E 对目标范围有**证据收敛**后，把 Observation digest 接入规划壳；LLM 不替代普查。  
- **参考**：框架 §阶段 D、 [加固路线图_要做的事.md](./加固路线图_要做的事.md) §六。

**门禁**：未满足框架 **§4.0 A+B** 类条件，不启动「以 LLM 为主力跑实网」项目。

---

## 2. 何为「乱搞」（本仓库明确反对）

- 无环境记录、无接口清单、无状态机，直接改 `packet_chain` 赌主按钮。  
- 把**新设示例资料**误解成「跑脚本前列表必须已有该企业名」。（资料用于从头核名/设立；仅当显式 `--enforce-listing-match` 时才强制列表正文含全称。）  
- 多标签、多脚本、并发请求压 9087。  
- 无 `census_run_record` 或等价 JSON 的「口头跑通」。  
- 把「云提交停点」理解成「可以自动点提交」。

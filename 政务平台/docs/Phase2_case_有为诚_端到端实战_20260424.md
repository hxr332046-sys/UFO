## 二、★ 发现的关键真 bug：signInfo 是动态签名

### 症状

同样的代码，在 case_有为风 (硬编码 `signInfo=-1607173598`) 跑 step 15 没直接 D0022，
但在 case_有为诚 跑 step 15 **立刻 D0022 越权访问**。

### 根因

`signInfo` 是**每个办件每次 load 都不同的动态签名**，不是工程手册之前写的"硬码常量"：

| 数据源 | signInfo |
|--------|----------|
| 硬编码 `SIGN_INFO_ESTABLISH` | `-1607173598` |
| 历史 mitm `step14_load_busidata.json`（有为风） | `-1506709975` |
| 今天 case_有为诚 step14 load 响应 | `-2013029225` |

三个值都不同。`phase2_bodies.py:build_basicinfo_save_body` 之前写死 `SIGN_INFO_ESTABLISH`，
save 时如果 load 返回值与之不一致 → 服务端判定伪造 → D0022。

### 修复（已落地）

**1. `@/UFO/政务平台/system/phase2_bodies.py:228` —** build 优先读 base 里 load 返回的 signInfo：
```python
body["signInfo"] = str(base.get("signInfo") or SIGN_INFO_ESTABLISH)
```

**2. `@/UFO/政务平台/system/phase2_protocol_driver.py:543-609` —** 新增两个 helper：
- `_apply_dynamic_sign_info(body, c)` — 用 `c.snapshot["last_sign_info"]` 覆盖 body.signInfo
- `_preload_component_sign_info(client, c, comp_url)` — save 前先 load 该组件拿最新 signInfo
- `_post_save_and_track_sign(..., preload_comp_url=...)` — 包装 save：preload + apply + post + 捕获新 signInfo

**3. `@/UFO/政务平台/system/phase2_protocol_driver.py:474` —** step14 load 成功后同步 `snapshot["last_sign_info"]`

**4. 所有 save step（15/16/18/19/20/22/23/24）—** 改用 `_post_save_and_track_sign(..., preload_comp_url="XxxInfo")`

### 回归测试（已落地）

`@/UFO/政务平台/phase1_service/tests/unit/test_phase2_bodies.py:32-54` — 新增 2 个测试：
- `test_sign_info_fallback_to_constant` — base 为空时 fallback 常量
- `test_sign_info_dynamic_from_base` — base 有 signInfo 时必须透传
- `test_sign_info_dynamic_historical_value` — 历史值 `-1506709975` 也要能透传

**当前总测试数：75 passed in 2.57s**（之前 73，新增 2 个 signInfo 铁律）。

## 四、未解问题（下一步攻坚方向）

### 4.1 step 16 MemberPost save — A0002 / D0003

**症状**：
- 开启 preload 前：D0003 服务器繁忙
- 开启 preload 后：A0002 服务端异常

**推测**（待 mitm 实抓验证）：
- preload MemberPost 返回的 signInfo 和 BasicInfo 一样（`-2013029225`），说明**整个办件 scope 共享 signInfo**，不是组件级
- MemberPost save 的 8-key body 可能**缺字段**：例如 `memberList[].cerNo` / `operatorInfo` / `confirmFlag` 等
- 或是 **save 顺序不对**：浏览器可能先 MemberPool/list 再 MemberPost save（我们目前是 16→17→18）

**建议路径**：
1. 打开 mitm，手工在浏览器里从"已有 BasicInfo save 成功"状态点击下一步 → 抓 MemberPost save 真实 body
2. 对比我们发的 8-key body 和真实 body 字段差异
3. 改 `build_memberpost_save_body`

### 4.2 step 18/19/20 D0003/D0009 — "未按正常顺序"

因 step 16 未成功 → 状态机停留在 MemberPost → 后续 save 全部报"顺序错"。  
修好 step 16 后，18/19/20 大概率连带修复。

### 4.3 step 21 SlUploadMaterial — TypeError

**位置**：`@/UFO/政务平台/system/phase2_protocol_driver.py:807`

```python
final["data"]["_upload_results"] = results   # ← final["data"] 为 None 时炸
```

**修复**：加 None 检查 —— 1 行改动：
```python
final.setdefault("data", {}) or final.update({"data": {}})
final["data"]["_upload_results"] = results
```
实际 `setdefault` 对 `None` 不生效，应该：
```python
if final.get("data") is None:
    final["data"] = {}
final["data"]["_upload_results"] = results
```

### 4.4 preload 副作用

部分组件（如 MemberInfo）单独 load 会 D0021，说明它们**只能在对应 list load 后**才能存在。  
当前 preload 对这些组件无效但不致命（保留 last_sign_info）。  
长期应分类：哪些组件支持 preload / 哪些不支持。

## 八、第二轮攻坚（当日下午 — step 16 深度逆向）

前面锁定 step 16 是下一个攻坚点，下午全力攻它。**已把 body 结构 100% 对齐 mitm 真实成功样本，但 D0003 剩余疑云需 mitm 实抓才能破解。**下面是完整攻坚链路。

### 8.1 发现：真实流程是 4 API 组合（不是单步 save）

对比 mitm 真实成功样本 `@/UFO/政务平台/dashboard/data/records/establish_save_samples/MemberPost__save.json:1` 和 `MemberBaseInfo__save.json:1` 的**时间戳**：

- `MemberBaseInfo save` ts=**1776911633**（先）— `compUrlPaths=["MemberPost","MemberBaseInfo"]`，cerNo **RSA 密文**
- `MemberPost save` ts=**1776911986**（后）— `compUrlPaths=["MemberPost"]`，cerNo **明文**

**真实浏览器流程**：
1. BasicInfo save 成功 → 切到 MemberPost 页面
2. UI 自动 `MemberBaseInfo load` → 拿到 49-key 空模板（含 fieldList + signInfo）
3. UI 在 MemberBaseInfo 子页编辑成员信息
4. `MemberBaseInfo save`（含 RSA cerNo）→ **服务端创建 member 记录分配 itemId**
5. 回 MemberPost 页面 → UI 自动 `MemberPool/list load` → 拿到 list[0] 含 itemId
6. 用户点保存 → `MemberPost save`（pkAndMem 里 4 个角色各一份 48-key 完整 member 对象，cerNo 明文）

我们原代码直接 MemberPost save → 跳过了 step 2/3/4 → 服务端没有 member 记录 → D0003/A0002。

### 8.2 重构：step 16 变为 4-API 组合操作

`@/UFO/政务平台/system/phase2_protocol_driver.py:616-700` 里新 `step16_memberpost_save` 内部依次：
1. `_preload_component_sign_info` → MBI load（拿模板）
2. `build_memberbaseinfo_save_body` + `establish_comp_op("MemberBaseInfo")` → MBI save
3. `step17_memberpool_list_load` → 拿 itemId / raw_member
4. `build_memberpost_save_body(raw_member)` → MemberPost save

**对 STEPS_SPEC 语义不变**：外部仍是 step 16 一步。

### 8.3 新增 `build_memberbaseinfo_save_body`

位置：`@/UFO/政务平台/system/phase2_bodies.py:236-343`

关键结构（基于 mitm 实录）：
- **48 顶层 keys**（load 返回 49 - `currentLocationVo` = 48）
- **linkData 5-key 干净版**（不继承 load 返回的 `busiCompComb/compCombArr/continueFlag`）
- **`cerNo` RSA 加密**（和 MemberPost 明文不同）
- **`postCode` 顺序 "FR05,WTDLR,LLY,CWFZR"**（和 MemberPost 的 "FR05,WTDLR,CWFZR,LLY" 相反）
- **`invType` = null**（不是 "1"，和样本一致）
- `sexCode` / `birthday` 从身份证号自动推断
- `personImgDto` 用已知可用 UUID 或 case 配置

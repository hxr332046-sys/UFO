# 阶段验收清单（设立登记：两阶段 + 办件状态）

> **问题背景**：仅「能连 CDP、能点按钮」不等于业务成功；须明确 **什么算第一阶段完成、什么算第二阶段完成、何时禁止进入下一步**。  
> **工程落点**：每次 `packet_chain` / `run_case_*` 运行结束后，查看输出 JSON 的 **`phase_verdict`**（`ufo.phase_verdict.v1`）与 **`acceptance`** 数组（含 `AC-PHASE1-*`、`AC-YUN-SUBMIT`）。

---

## 1. 第一阶段 — 名称登记可接续

| 验收项 | 怎样算「通过」 | 怎样算「未通过 / 未知」 |
|--------|----------------|-------------------------|
| **核名结果** | 系统在名称申报流程中给出 **可用** 结论，并按平台规则生成后续材料（如告知类文件；以实际页面为准）。 | 提示 **重名 / 违规 / 不可用** 等，未拿到可用结论。 |
| **我的办件出现对应用件** | 在 **我的办件** 列表中，能检索到与 **拟设主体名称一致或唯一对应** 的一条（企业名称列）。 | 列表中 **没有** 该名称；或只有其它主体（如截图中仅有「陈飞李…」而无「李陈梦…」）。 |
| **当前状态（人工读表）** | 以列表 **「当前状态」** 为准：常见为 **填写中**（可点「继续办理」进入第二阶段）。 | **不予受理 / 已退回** 等需先处理完再谈自动化。 |

**自动化可选检查**（仅当显式加 **`--enforce-listing-match`**）：

- 在 **我的办件列表页**时：若页面正文**不含** `case` 中的 **`company_name_full`**，则 **立即中止**（用于「续办前确认列表里就是这一单」等排障场景）。  
- **默认不加此参数**：资料为**新设从头**示例时，**不要求**列表里已出现该企业名（核名前/未生成办件时列表里没有是常态）。

已在 **core.html** 续跑、未经过列表页时：**不**自动验列表，结论可能为 `unknown`，需人工对照上表与名称登记进度。

---

## 2. 第二阶段 — 长表单至云提交停点

| 验收项 | 怎样算「通过」（本仓库任务完成） | 未通过 |
|--------|----------------------------------|--------|
| **进入设立主流程** | 曾进入 **`core.html`** 填报/材料步骤（技术项 `AC-CORE-REACHED`）。 | 始终停在门户 / 名称子应用 / 列表页。 |
| **云提交停点** | 页面文案出现 **云提交 / 云端提交** 等停点探测命中（`AC-YUN-SUBMIT`、`reached_yun_submit`），**脚本不点击提交**。 | 未探测到上述文案。 |

**与列表「当前状态」的关系**：列表中的 **填写中** 只说明 **尚有步骤未完成**；第二阶段 **完成** 以本仓库约定为准：**出现云提交相关界面/文案即停**，不要求列表状态已变为「审核通过」（后者多为提交后经办端状态）。

---

## 3. 读结果时的顺序（避免再「只会干」）

1. **先看终端**：`run_case_rongxian_to_yun_submit.py` 结束时会打印 **`=== 业务结论（必读）===`**，并返回**非零退出码**表示业务未闭环（勿只看「脚本跑完」）。  
2. 打开 **`phase_verdict`**：先看 **第一阶段 `status`**，再看 **第二阶段**。  
3. 打开 **`acceptance`**：关注 **`AC-PHASE1-COMPANY-ON-LIST`**、**`AC-YUN-SUBMIT`**，勿仅用 **`AC-REACH-FLOW`** 判断业务成败（该条为技术里程碑，已在注释中说明）。  
4. 打开 **`framework_notes.issues`**：阶段结论会以 **【阶段结论】** 前缀写入。  
5. 仍须结合 **`steps`** 与截图目录中的 **`blocker_evidence`** 做人工复核。

### `run_case_rongxian_to_yun_submit.py` 进程退出码

| 码 | 含义 |
|----|------|
| **0** | 第二阶段达成：已探测到云提交文案停点（未点提交）。 |
| **3** | **第一阶段明确未过**（如列表无案例企业名、门禁中止）。 |
| **4** | 未达第二阶段停点，或第一阶段为 **unknown**（须人工对照「我的办件」）。 |
| **2** | 前置错误（如缺 case 文件、未写出 JSON）。 |

## 4. 与办件列表 UI 的对应（你提供的截图）

- **办理时间 / 企业名称 / 办理事项 / 当前状态 / 操作** 列：自动化应优先用 **企业名称** + **当前状态** 判断「是不是这一单」「能不能继续」，而不是盲目点第一个「继续办理」。  
- **填写中**：通常表示尚未办完表单，可 **继续办理** 进入向导。  
- **审核通过**：多表示 **该阶段或该办件已审结**，与「是否已到云提交停点」不是同一概念，须按 §2 区分。

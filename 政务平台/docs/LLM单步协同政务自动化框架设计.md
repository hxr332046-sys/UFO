# LLM单步协同政务自动化框架设计

## 1. 设计目标

本框架不再以“整段脚本自动跑完整业务链”为目标，而是以 **LLM 驱动的单步协同 + 每步确认** 为核心模式。

适用场景：

- 政务平台存在强上下文依赖
- 每一步都可能出现动态提示、资格确认、材料告知
- 平台对连续自动脚本敏感，存在风控或封禁风险
- 业务推进需要人的主观判断与最终确认

核心原则：

- **先观测，后动作**
- **一次只推进一步**
- **每步执行后必须验结果**
- **高风险动作必须人工确认**
- **失败回到状态，不盲目重试**

---

## 2. 框架定位

### 2.1 不再追求的能力

以下目标不再作为主目标：

- 从入口页一键自动跑到业务终点
- 通过长循环脚本连续点击直至到达下一阶段
- 把复杂提示、确认、资格弹窗、人工判断动作都交给脚本处理

### 2.2 新的主目标

本框架的主目标是：

- 识别当前页面所处业务状态
- 读取当前页面的提示、按钮、弹窗、表单摘要
- 由 LLM 规划“当前最合理的下一步”
- 对高风险动作要求人工确认
- 执行单步后再次验证页面是否真正进入新状态
- 把所有状态变化落盘，形成可追溯证据

---

## 3. 总体架构

```text
用户
  ↓
LLM 协同控制器
  ↓
状态探测层（Perception）
  ↓
决策层（Policy / Planner）
  ↓
单步执行层（Actuator）
  ↓
结果验证层（Verifier）
  ↓
日志 / 证据 / 状态持久化
```

### 3.1 用户角色

用户负责：

- 确认高风险动作是否执行
- 处理平台要求的人类判断动作
- 完成资格确认、身份确认、最终提交等敏感节点
- 对页面实际表现进行补充说明

### 3.2 LLM 角色

LLM 负责：

- 理解当前状态
- 识别阻塞项
- 规划单步动作
- 判断动作风险
- 解释为什么这样做
- 在失败后做归因并选择下一种策略

### 3.3 浏览器/CDP 角色

CDP 层只负责：

- 读取当前页面信息
- 执行一条小动作
- 返回动作结果和页面变化

它不负责：

- 自行规划整链
- 长循环重试
- 越权点击敏感操作

### 3.4 API/协议层角色

API 层只负责：

- 字典与枚举探测
- 低风险协议校验
- 保存结构辅助分析
- 为页面填写提供候选值支持

它不负责：

- 单独主导真实业务链推进
- 替代页面中的资格确认和人工判断

---

## 4. 分层设计

## 4.1 状态探测层（Perception）

职责：统一读取当前页面的真实状态。

输入：

- 当前 URL / hash
- 页面正文文本
- 可见按钮
- 可见弹窗/提示
- Vue 组件摘要
- 最近接口尾包（可选）

输出建议统一为结构化状态对象：

```json
{
  "page_type": "guide_base",
  "href": "...",
  "hash": "...",
  "blocking_prompts": ["请确认您属于上述人员范围"],
  "visible_buttons": ["下一步", "关闭", "确定"],
  "form_summary": {
    "entType": "1100",
    "distCode": "",
    "address": ""
  },
  "confidence": 0.92
}
```

建议识别的主状态：

- `portal`
- `enterprise_zone`
- `declaration_instructions`
- `guide_base`
- `my_cases`
- `name_check`
- `core_basic_info`
- `face_or_sms_gate`
- `unknown`

---

## 4.2 决策层（Policy）

职责：规定当前状态下哪些动作可自动执行，哪些必须人工确认。

建议输出三类动作：

- `safe_auto`
- `need_confirm`
- `forbidden`

示例：

### safe_auto

- 读取弹窗内容
- 聚焦输入框
- 展开下拉框
- 读取可选项
- 预填低风险字段
- 抓取当前页面证据

### need_confirm

- 点击“继续办理”
- 选择企业类型
- 确认资格提示
- 点击“保存并下一步”
- 从我的办件进入下一阶段

### forbidden

- 最终提交
- 云提交
- 发起验证码认证
- 代替用户做人脸/实人认证
- 自动确认具有法律效力的声明

---

## 4.3 规划层（Planner）

职责：只规划 **下一步**，而不是整条链。

每轮输出建议固定结构：

```json
{
  "current_state": "guide_base",
  "goal_of_next_step": "完成资格确认并使guide/base进入可填写状态",
  "proposed_action": "读取资格提示后由用户确认，再执行地址字段预填",
  "why": "当前平台存在阻塞提示，直接推进下一步会失败",
  "risk_level": "medium",
  "needs_user_confirmation": true,
  "expected_result": "guide/base 组件恢复可交互或进入名称核查页",
  "fallback_if_failed": "重新采集弹窗与Vue状态，判断是否仍在资格阻塞"
}
```

规划层必须遵守：

- 一次只输出一步
- 不跨多个状态做长链规划
- 不能假设上一步一定成功
- 每次都必须读取最新状态后再决定

---

## 4.4 执行层（Actuator）

职责：执行一条明确且可验证的单步动作。

支持动作建议限定为：

- `read_state`
- `click_one`
- `fill_one_field`
- `choose_one_option`
- `wait_transition`
- `capture_evidence`

不建议保留的执行模式：

- `while True` 式连续推进
- “直到进入某页为止”的自动点击循环
- 多步动作打包执行

---

## 4.5 验证层（Verifier）

职责：判断执行结果是否真的达成预期。

验证信号包括：

- URL/hash 是否变化
- 页面是否出现新按钮/新提示
- 组件 VM 是否拿到新值
- 是否进入新的业务状态
- 是否仍停留在原阻塞点

示例：

- 预期进入 `name_check`
- 实际仍停在 `guide_base`
- 说明动作失败，应进入失败归因，而不是重复点下一步

---

## 5. 状态机设计

状态机不按脚本步骤设计，而按 **业务观察状态** 设计。

建议主状态：

- `S00_PORTAL`
- `S05_ENTERPRISE_ZONE`
- `S06_DECLARATION_NOTICE`
- `S08_GUIDE_BASE`
- `S10_NAME_CHECK`
- `S20_MY_CASES`
- `S30_CORE_BASIC_INFO`
- `S90_BLOCKED_PROMPT`
- `S99_UNKNOWN`

每个状态应定义：

- 可观测特征
- 常见阻塞项
- 可提议动作
- 必须人工确认的动作
- 成功判定
- 失败回退策略

---

## 6. 与现有代码的关系

## 6.1 `packet_chain_*`

重新定位为：

- 页面探测片段库
- 恢复动作库
- 已验证 JS 经验库

例如可复用内容：

- `GUIDE_BASE_AUTOFILL_V2`
- `S08_STATE_PROBE_JS`
- `CLICK_RECOVERY_STUCK_JS`
- `S08_GUIDE_FLOWSAVE_JS`

但它们不应继续作为“整链自动推进脚本”的主体。

## 6.2 `run_phase1_from_case.py`

重新定位为：

- 第一阶段的协同控制器原型

后续应改造成：

- 读取当前状态
- 输出推荐下一步
- 接收确认
- 执行单步
- 记录结果

而不是从入口到目标页的一次性推进器。

## 6.3 `icpsp_api_client`

保留为：

- 协议探针
- 字段辅助校验
- 字典值参考
- 请求结构分析

不作为业务主链驱动器。

---

## 7. 日志与证据模型

框架落盘记录的重点不应是“脚本执行到哪一行”，而应是：

- 当前状态是什么
- 识别到哪些阻塞项
- 提议了什么动作
- 动作是否需要人工确认
- 执行后是否达成预期状态
- 如果失败，失败原因是什么

建议单轮日志结构：

```json
{
  "round": 7,
  "state_before": "S08_GUIDE_BASE",
  "blocking_items": ["资格提示", "地址未落值"],
  "recommended_action": "用户确认资格提示后执行地址预填",
  "needs_confirmation": true,
  "action_result": "executed",
  "state_after": "S08_GUIDE_BASE",
  "success": false,
  "reason": "仍未离开guide/base"
}
```

---

## 8. 风控与边界

本框架必须长期坚持以下边界：

- 不做长时间无反馈连续点击
- 不绕过系统提示与资格确认
- 不自动执行最终提交
- 不自动处理实人/短信/认证闸门
- 不在状态未知时盲目执行下一步
- 不把“读到旧页签”误判为“进入新状态”

---

## 9. 最小可落地版本（MVP）

优先落地三个能力：

### 9.1 页面状态探测器

目标：统一输出当前页的页面类型、按钮、弹窗、表单摘要。

### 9.2 单步动作协议

目标：统一定义最小动作集：

- 读取
- 点击一次
- 填一个字段
- 选择一个选项
- 等待一次跳转
- 验证一次结果

### 9.3 协同回合控制器

目标：每次只做一轮：

- 读状态
- 提建议
- 等确认
- 执行动作
- 验证结果
- 写日志

---

## 10. 当前阶段的实施顺序

建议按以下顺序推进：

1. 统一页面状态探测模型
2. 抽象单步动作接口
3. 实现“建议动作 + 等确认”的回合控制器
4. 将 `guide/base` 先纳入协同模式
5. 再扩展到 `my_cases -> continue` 和 `core` 表单阶段

---

## 11. 最终判断标准

框架是否有效，不看“能不能一键跑通”，而看：

- 是否能稳定识别当前状态
- 是否能把阻塞原因说清楚
- 是否能给出正确的下一步建议
- 是否能在人工参与下稳定推进业务
- 是否能降低误操作与风控概率

如果这些都达成，那么这个框架就是对的。

## 二、D0022 的"三层洋葱"

这个错误之所以折腾人，是因为它**语义模糊**——不管是 header、body、SESSION、还是"魔数"有问题，服务端统一返回 `D0022 越权访问`，没有任何提示。

```
┌─────────────────────────────────────────────────┐
│        第 1 层：浏览器指纹头组（Header）         │   → 缺任一 sec-ch-ua / Sec-Fetch-*  D0022
├─────────────────────────────────────────────────┤
│       第 2 层：请求 body 结构（Schema）          │   → 多 / 漏字段推 SESSION 到错分支  D0022
├─────────────────────────────────────────────────┤
│       第 3 层：signInfo 魔数（Magic Number）     │   ★ 最隐蔽一层，真凶藏在这
└─────────────────────────────────────────────────┘
```

## 三、核心方法论：**"样本即真理" 四步法**

面对"服务端黑盒 + 只有一个糊涂错误码"的场景，**不要猜、不要试**，只做一件事：**逼近真实浏览器在生产环境下发的 byte-exact 请求**。

### 第 1 步 · 离线 diff 铺路（不发请求）

先把**真实浏览器**（从 mitm/Fiddler 备份抓的）和**驱动器**各自构造的请求 body 做**递归字段级 diff**。每一条不同都是嫌疑。绝对不先发请求——发一次等 5 分钟冷却，代价巨高。

工具：`system/_offline_diff_steps_1_4.py`、`system/_offline_deep_diff_step5.py`。

```python
# 用拦截 client 捕获 driver 真实发出的 body（不需要发请求）
class _InterceptClient:
    def post_json(self, path, body):
        raise _IntBody("POST", path, body)

driver_body = _intercept_body(step_fn, ctx)
sample_body = mitm_parse(backup_path, api_path)
diff = deep_diff(driver_body, sample_body)   # 递归 dict/list
```

### 第 2 步 · 分层修正（Header → Body 结构 → Body 数据 → 魔数）

按"表层→深层"顺序修。不要跳，否则状态污染难复现。

- **表层：头**。只要样本里有 `sec-ch-ua-mobile`，驱动器就必须带。不讨价还价。
- **中层：body 结构**。驱动器**多**发一个字段会把服务端 SESSION 推进到错误的分支。**漏**发一个字段会让服务端无法绑定资源（如 `itemId`）。**结构先于数据**。
- **深层：body 数据值**。多个样本**同一个字段取同一个值**，而驱动器按业务逻辑算出来的不同——这就是"魔数"嫌疑（见下文 `signInfo`）。

### 第 3 步 · 单变量精准探针（有前提才发）

确认 diff 报告可信之前，**不允许发任何请求**。  
确认之后，每次只改一个变量，步间 ≥ 15 秒（躲避服务端限流 `D0029`）。

我做的是 `_single_shot_d0022_probe.py --field X --value Y`（已随突破完成而删除）。

### 第 4 步 · 确权实验：浏览器 fetch + 当前用户 Auth

最决定性的判据：**让同一个浏览器、同一个登录用户、发驱动器构造的 body**。
- 浏览器过 + driver 不过 → 纯 HTTP 指纹问题（TLS/HTTP2 层）
- 浏览器也不过 → 问题就是 body 本身，**不是**鉴权/指纹

我在 `_single_browser_probe.py` 里用 CDP `Runtime.evaluate` 注入 `fetch()`，当前用户发 driver body。**浏览器也 D0022** 才锁定"body 自身有问题"这一分支。

## 四、三层真凶的详细证据

### 真凶 1：浏览器指纹头组（`Header` 层 D0022）

服务端严格校验这些"浏览器自有"的 header 是否完整：

```python
_BROWSER_LIKE_HEADERS = {
    "User-Agent": "...Edg/...",            # Edge UA 必须是 Edge，不能是 Chrome
    "sec-ch-ua": '"Chromium";v="137", ...',
    "sec-ch-ua-mobile": "?0",
    "sec-ch-ua-platform": '"Windows"',
    "Sec-Fetch-Site": "same-origin",
    "Sec-Fetch-Mode": "cors",
    "Sec-Fetch-Dest": "empty",
    "Accept-Encoding": "gzip, deflate, br, zstd",
    "Accept-Language": "zh-CN,zh;q=0.9",
}
```

缺任一 → D0022。Python `requests` 默认不带这些——**必须手工注入**。

### 真凶 2：body 结构不对齐（`Schema` 层 D0022）

用离线 diff 发现，driver 和真实浏览器样本在 step 2/3 有**结构性**差异。**即使服务端对这些 step 返回 200**（服务端容忍未知字段），也会在**后续 step 5** 反应成 D0022——因为 SESSION 已被推进到错误分支。

| step | driver 错误做法 | 真实浏览器 | 影响 |
|------|----------------|-----------|------|
| step 2 `loadCurrentLocationInfo` | 多发 `extraDto` 整个对象 | 根本没这字段 | SESSION 进入"带扩展参数"分支 |
| step 2 | `flowData.vipChannel = "null"` | 不带此字段 | 服务端误判 VIP 通道 |
| step 2 | `linkData.continueFlag = "continueFlag"` | 只 `{token:""}` | 服务端当作"续办"，去查不存在的 busiId |
| step 3 `loadBusinessDataInfo` | **漏** `itemId: ""` | 有 `itemId: ""` | 无法绑定业务组件实例 |

修法：让 `step_load_current_location` 只发最小三字段 `flowData`，`linkData` 只带 `{token:""}`，**整个去掉 `extraDto`**。在 `step_namecheck_load` 顶层补 `itemId: ""`。

### 真凶 3：`signInfo` 魔数（最深一层 D0022）

**这是最让我拍案的一层。**

step 5 `operationBusinessDataInfo#first` 的 body 里有个 `signInfo` 字段。按"代码常理"我用"name+nameMark+organize 的 Java hashCode"动态算，**任何值**都 D0022。

然后我做了递归 diff，发现 mitm 备份里 **4 条首次保存样本，`signInfo` 全部是 `-252238669`**。

> **服务端把 `signInfo` 当"已知合法魔数"校验——不是对 body 内容做 hash，也不是和用户挂钩的 token，就是前端硬编码写死的一个数。**

前端开发者可能图省事直接 `return -252238669`；服务端也只白名单这一个值。动态算任何值都不在名单里，所以拒。

修法：

```python
# system/phase1_protocol_driver.py:DriverContext.from_case()
# signInfo：服务端把它当"已知合法魔数"校验（不依赖 body 内容 / 不依赖用户）。
# mitm 备份 4/4 首次保存样本的 signInfo 全部是 "-252238669"。
# 之前 driver 用动态 Java hashCode，任意变化都触发 D0022 越权访问。
# 修复：直接锁定为该固定值。
ctx.sign_info = -252238669
```

**这就是用户强调的那句话"整个流程是有上下文状态管理的"在数据层的体现——有些字段根本不是业务值，而是会话约定。**

## 七、系统性的反制：防回归清单

1. **`signInfo` 锁定为魔数**：任何 case、任何字号、任何用户都用 `-252238669`。如果未来服务端轮换，只改一个常量。
2. **`_BROWSER_LIKE_HEADERS` 禁删**：不管看起来多"冗余"，少一个就 D0022。
3. **step 1-4 body 禁自由发挥**：以 mitm 备份样本为真理，离线 diff 必须为空才能上线。
4. **每次改驱动器后先跑 `_offline_diff_*` 离线校验**：不出门先照镜子。
5. **限流保护**：driver 已内置步间 0.9 秒节流 + `D0029` 熔断。

## 八、元方法论：面对黑盒服务端的三个原则

这是我在这次攻坚中凝练的三条**可迁移**原则，以后遇到其他政务/金融/银行黑盒也一样有效：

### 原则 A：**"样本即合约"（Samples-are-the-Contract）**

没有文档的服务端，它唯一可靠的契约就是"它接受过的样本"。离线 diff 它和你之间的差异，逐点拉平——不要依靠"合理推测"。

### 原则 B：**结构先于数据，数据先于魔数**

修请求体时按这个顺序：
1. 字段键（schema）对不对
2. 字段值的"业务语义"对不对
3. 字段值的"具体数值"对不对（魔数层）

跳层修，必然来回走。

### 原则 C：**每一次"不经思考的请求"都是债务**

黑盒服务端都有限流和风控。一次错误尝试 = 5 分钟等待 + 风控画像加深。**离线 diff 是最重的武器**，因为它让你在 0 代价下完成 95% 的定位。

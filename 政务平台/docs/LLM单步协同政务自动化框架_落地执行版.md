# LLM单步协同政务自动化框架（落地执行版）

基于：`docs/LLM单步协同政务自动化框架设计.md`  
用途：直接指导“LLM + CDP + 协议辅助”的实网执行，不走纯脚本黑盒。

## 1. 执行总原则

- 一次只做一步：**观测 -> 动作 -> 验证 -> 记录**。
- 高风险动作必须明确确认（或由用户手动完成）。
- 不跨阶段跳步，不以“长循环直到成功”为策略。
- 每次失败都必须沉淀为可复用经验条目。

## 2. 标准状态机（实网）

- `my_space`：我的办件列表
- `guide_base`：名称登记入口页（企业类型/是否已申请/地区）
- `name_check`：名称核查页
- `name_success`：申报成功页（第一阶段终点）
- `unknown`：无法识别，必须先观测再动作

## 3. 单步协议

每一步都输出结构化记录：

```json
{
  "state_before": "...",
  "goal": "...",
  "action": "...",
  "result": "...",
  "state_after": "...",
  "evidence_files": []
}
```

## 4. 固化前置动作（每轮必做）

1. 重装监听（刷新后必须重装）；  
2. 清理上下文污染（`_topnet_flowParams` / `continueFlag`）；  
3. 探测当前路由与可交互按钮；  
4. 仅当状态确认后再执行下一动作。

## 5. 第一阶段执行模板

### 步骤 A：进入名称登记状态
- 从入口页到 `guide_base` 或 `name_check`。
- 如果被待办干扰，先解绑再重试，不直接续办旧单。

### 步骤 B：填写并提交名称
- 设置行政区划、字号、行业、组织形式；
- 勾选“我已阅读并同意”；
- 点击“保存并下一步”；
- 记录请求与响应。

### 步骤 C：验收
- 命中 `name_success` 或同等成功提示；
- 回到“我的办件”确认新增记录；
- 仅此判定第一阶段成功。

## 6. 失败处理策略

### A0002
- 刷新一次后重试该步；
- 记录请求体与响应体；
- 连续失败则转人工一步并继续监听。

### 路由偏转到旧草稿
- 先清 flowParams + continueFlag；
- 调 `flow-control` 重建新建态；
- 再回目标步骤。

### 删除干扰草稿失败
- 记录 before/after 接口返回；
- 不在该步死循环；
- 允许人工删除后继续主流程。

## 7. 与“纯脚本”边界

- 本框架不追求全自动闭环提交；
- 核心目标是“LLM可解释地推进每一步，并留存证据”；
- 脚本只是执行器，LLM负责状态理解与决策。

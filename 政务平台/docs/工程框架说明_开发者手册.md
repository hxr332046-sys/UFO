## 1. 30 秒认识这个项目

**做什么**：把广西经营主体登记平台（税务总局 ICPSP 底座）的**企业设立办理**协议化，替代浏览器自动化。

**为什么**：浏览器自动化脆、慢、抓不到加密细节；协议化后可在 5 秒完成以前要 5 分钟的流程，且可 LLM 调度。

**已达成**：
- **Phase 1 名称登记**：7 步完成，拿到 `busiId` + `nameId`（约 5-10 秒）
- **Phase 2 设立登记**：25 步到达云提交停点 `PreSubmitSuccess`（约 15-30 秒）
- **27 个 REST API** + **28 个 LLM function tools** + **26 个 CLI 命令**

**还未做**：电子签章（CA 证书依赖）、跨区划批量推广验证。

**当前状态（2026-04-24）**：框架骨架完整、铁律已锁、测试 73 个通过、单一样本全链路已打通。**下一步重心是跨案例通用性验证**。

## 2. 10 分钟跑起来

### 前置

- Windows + Python 3.11
- Chrome Dev（CDP 端口 9225 用于扫码登录）
- 已有一个案例 JSON（参考 `@/UFO/政务平台/docs/case_有为风.json:1-68`）

### Step 1：装依赖

```powershell
# 项目已预装在 .venv-portal
# 若需重建：
python -m venv .venv-portal
.\.venv-portal\Scripts\pip install -r requirements-portal.txt
.\.venv-portal\Scripts\pip install pytest
```

### Step 2：启动 FastAPI

```powershell
.\.venv-portal\Scripts\uvicorn phase1_service.api.main:app --host 127.0.0.1 --port 8800
```

- Swagger UI: http://127.0.0.1:8800/docs
- 端点总览: http://127.0.0.1:8800/

### Step 3：获取登录 token（首次）

```powershell
# 方式 A：扫码（首次必做）
python -m phase1_service.cli auth-qr-start
# 打开返回的 base64 图片 → 粤商通扫码 → 轮询：
python -m phase1_service.cli auth-qr-status --sid <返回的sid>

# 方式 B：静默续期（之前扫码过且 cookie 未过期）
python -m phase1_service.cli auth-ensure
```

Token 被自动保存到 `@/UFO/政务平台/packet_lab/out/runtime_auth_headers.json`，后续所有 API 自动使用。

### Step 4：跑一个冒烟流程

```powershell
# 名字预检
python -m phase1_service.cli precheck 有为风

# Phase 1：7 步拿 busiId
python -m phase1_service.cli register --case docs/case_有为风.json

# Phase 2：25 步到云提交停点
python -m phase1_service.cli phase2-register --case docs/case_有为风.json
```

### Step 5：跑测试确认环境

```powershell
python -m pytest phase1_service/tests -q
# 期望：73 passed in ~2.5s
```

✓ 跑通说明环境 OK。

## 4. 三层架构与一个请求的完整旅程

```
┌─────────────────────────────────────────────────────────────┐
│ 调用方（3 个入口都走同一套代码）                               │
│  - Swagger UI / 浏览器                                       │
│  - CLI (python -m phase1_service.cli ...)                   │
│  - LLM agent（通过 function calling）                        │
└───────────────────────────┬─────────────────────────────────┘
                            │ HTTP
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ Layer 1: FastAPI Router (phase1_service/api/routers/)        │
│  - 处理 HTTP 参数绑定（Pydantic）                             │
│  - 调 core/ 层的业务函数                                       │
│  - 不做业务逻辑，不直接调 system/                              │
└───────────────────────────┬─────────────────────────────────┘
                            │ Python 调用
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ Layer 2: Core Adapter (phase1_service/api/core/)             │
│  - driver_adapter.py / phase2_adapter.py                    │
│  - 参数归一化 + 错误码分类 + 幂等缓存 + session 自愈          │
│  - 调 system/ 的 driver.run() 或 step 函数                    │
└───────────────────────────┬─────────────────────────────────┘
                            │ Python 调用
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ Layer 3: Protocol Driver (system/)                           │
│  - phase1/2_protocol_driver.py : 每个 step 一个函数           │
│  - phase2_bodies.py : body 构造器                           │
│  - phase2_enums.py : 中文→code                              │
│  - icpsp_crypto.py : AES/RSA                                │
│  - icpsp_api_client.py : 唯一 HTTP 出网口                    │
└───────────────────────────┬─────────────────────────────────┘
                            │ HTTPS + 加密
                            ▼
                 广西政务平台 / ICPSP 底座
```

### 一个请求的完整旅程（以 `POST /api/phase2/register` 为例）

```
1. client 发 POST {case, stop_after: 14}
2. FastAPI 路由到 @/UFO/政务平台/phase1_service/api/routers/phase2_register.py
3. router 里 Pydantic 校验 body，调 phase2_adapter.register(...)
4. adapter 干的事：
   a. 查幂等缓存（sha1(case + busi_id + stop_after)）
   b. 未命中 → 调 drv.get_steps_spec() 拿 25 步清单
   c. 创建 ICPSPClient（自动装 Authorization + session cookies）
   d. 循环每步：调 step_N(client, ctx) → 解码 code/resultType
   e. 命中 D0022/A0002 → 分类错误 + 带 hint 返回
   f. session 被踢（A0005）→ 尝试 session_recovery 自愈
   g. 成功 → 把结果入幂等缓存
5. driver 里每步：
   a. 从 ctx 取 case 字段
   b. 调 phase2_bodies.build_xxx_body(...) 造 body
   c. 调 client.post() 实发
   d. ICPSPClient 内部：加 Authorization header + AES 加密 body.data
6. 响应逐层回升：driver → adapter → router → client
```

### 为什么要分三层

- **driver 层**：可独立跑（`python system/phase2_protocol_driver.py --case xxx.json --stop 14`），不依赖 FastAPI，便于调试/逆向
- **core 层**：业务语义（幂等/重试/session自愈/错误分类），不依赖 HTTP 框架
- **router 层**：只管 HTTP 绑定和 Pydantic，轻

## 5. 关键术语与心智模型

| 术语 | 含义 | 何时用 |
|------|------|--------|
| `busiId` | 办件 ID（Phase 1 创建时分配，贯穿整个申请） | 所有需要"在具体办件里做事"的 API |
| `nameId` | 名称 ID（Phase 1 step 8 `name/submit` 后分配） | Phase 2 所有接口的 `flowData.nameId` |
| `entType` | 企业类型代码（4540=个独 / 1100=有限 / 1151=有限(自然人独资) / 8000=合伙）| 每个 API 都需要 |
| `busiType` | 业务类型（01=设立 / 02=变更 …） | Phase 2 默认 `02_4` |
| `signInfo` | 签名校验魔数，**Phase 1 用 `-252238669` / Phase 2 establish 用 `-1607173598`** | 写操作 body 必带 |
| `opeType` | 操作类型：`save` / `special` / `load` | save=推进状态机，special=上传证明材料 |
| `currCompUrl` | 当前组件 URL（如 `BasicInfo` / `MemberPost`） | `flowData.currCompUrl` + `linkData.compUrl` |
| `compUrlPaths` | 已走过的 step 路径数组，`busiCompUrlPaths` 是 URL-encoded 版 | 铁律：save 时 `busiCompUrlPaths="%5B%5D"`（空数组） |
| `case` | 一个申请的完整业务数据 JSON，驱动所有 API | 参考 `@/UFO/政务平台/docs/case_有为风.json:1-68` |
| `sysParam` | 平台全局 958 条系统参数（aesKey、RSA 公钥、各种配置） | 密钥从这里读 |
| `aesKey` | 请求体 `data` 字段的 AES 加密密钥（平台写死 `topneticpsp12345`） | 所有 POST 都要 |

### 核心心智模型：**"平台状态机" vs "客户端推进"**

- 平台维护一个办件的状态机（25+ 个状态）
- 每次 `opeType=save` 就是**推进状态机一格**
- `load` 是读当前位置（不改状态）
- `currCompUrl` 是当前在哪一格；`busiCompComb` 是已完成格子的路径
- **铁律**：推进前必须先 `load` 当前位置（否则越权）

## 6. Phase 1 / Phase 2 协议链全景

### 6.0 先看业务上的两阶段，再看协议步骤

> 统一口径见 `@/UFO/政务平台/docs/流程.md:1`。后续所有 API、CLI、Runner、日志归档都应优先按**业务阶段**理解，再下钻到协议 step。

业务上，本项目只有两阶段：

- **阶段一：名称信息（核名）**
- **阶段二：信息填报**

但当前代码实现上，二者和 driver 文件名**并不是严格一一对应**：

- `phase1_protocol_driver.py`
  - 负责名称阶段前半段
  - 重点是名称预检、校验、保存、拿到 `busiId`

- `phase2_protocol_driver.py` 的 **step 1-9**
  - 技术上放在 Phase 2 驱动器里
  - 业务上其实仍属于**第一阶段：名称信息**
  - 包含 `NameSupplement`、`NameShareholder`、`name/submit`、`NameSuccess`

- `phase2_protocol_driver.py` 的 **step 10+**
  - 才真正进入**第二阶段：信息填报**
  - 从 `matters/operate` 进入 establish，再推进到 `PreSubmitSuccess`

因此，推荐你始终按下面三层理解系统：

1. **业务阶段**：名称信息 / 信息填报
2. **页面流程**：基本信息 / 成员架构 / 成员信息 / 补充信息 / 税务 / 材料 / 领照 / 云帮办
3. **协议步骤**：`STEPS_SPEC` / `get_steps_spec()` 中的具体 step

### Phase 1 · 名称登记（7 步）

```
Step 1  name/check                          名字查重（检查是否被占用）
Step 2  tools/getCacheCreateTime            拿缓存时间戳（反爬必备）
Step 3  cachedate/get                       拿字典缓存
Step 4  name/loadInit                       页面初始化
Step 5  common/queryCodeKV                  拿 KV 配置
Step 6  login/checkToken                    验证登录态
Step 7  name/handleTaskAndLoadFlowLink  ★   创建办件，返回 busiId
```

完整源码：`@/UFO/政务平台/system/phase1_protocol_driver.py:1-700`

## 7. 铁律清单（碰不得的硬约束）

> 这些是从几十次踩坑里总结出来的"血的教训"。**改代码前读一遍**，每条都有测试在 `phase1_service/tests/` 锁定。

### 7.1 加密与签名

- **`signInfo` 是动态签名（★ 2026-04-24 订正）**：不是常量！
  - Phase 1 的 signInfo 恒定 `-252238669`（这是 Phase 1 特殊）
  - **Phase 2 establish 每个办件、每次 load 返回不同值**（历史值 `-1506709975` / 实测 `-2013029225`）
  - `SIGN_INFO_ESTABLISH = -1607173598` 只是 **fallback**
  - **save 必须回传最近一次 load/save 返回的值**，否则 D0022 越权
  - Driver 已用 `_post_save_and_track_sign(..., preload_comp_url=X)` 自动化此逻辑
  - 完整记录：`@/UFO/政务平台/docs/Phase2_case_有为诚_端到端实战_20260424.md:1`
- **`aesKey="topneticpsp12345"`** 是平台写死的，不从 sysParam 读（sysParam 里的 aesKey 是**另一层**用途）
- **RSA 加密用平台 `numberEncryptPublicKey`**，结果始终是 172 字符 Base64
- **AES 加密的是 body.data，不是整个 body**。header/linkData/flowData 明文

### 7.2 字段大小写

- `cerno` **小写 n**（大写 `cerNo` 会触发 A0002 — step 21 SlUploadMaterial 踩过）
- 详见 `@/UFO/政务平台/phase1_service/tests/unit/test_phase2_bodies.py:120`（已有测试锁定）

### 7.3 URL 编码

- `busiAreaData` 必须是 `JSON → URL-encode` 两次（不是普通 JSON 字符串）
- `busiCompUrlPaths` save 时固定 `"%5B%5D"`（空数组 URL-encoded），load 时由服务端返回
- `compUrlPaths` 是**数组**（明文 JSON 数组），`busiCompUrlPaths` 才是 URL-encoded

### 7.4 状态机推进

- **save 前必须先 load 当前 compUrl**（否则 D0022 越权）
- 详见 `@/UFO/政务平台/docs/突破D0022越权_协议层字段合同方法论_20260422.md:1`
- step 13 (`YbbSelect load`) 若返回 D0021 是**正常**，不中断（optional=True）
- step 17 (`MemberPool list`) 失败也 optional，不阻塞后续

### 7.5 session / token

- Authorization 是 **32-hex**（小写）；过期后 A0005 需扫码重获
- session cookies 关键 3 个：`JSESSIONID`、`SESSIONFORTYRZ`、`user_token_uuid`
- 扫码登录后必须调 `phase2_session_recover` 从 CDP 同步 cookies，否则协议接口 401
- 详见 `@/UFO/政务平台/docs/SSO自动登录技术经验.md:1`

### 7.6 STEPS_SPEC 单一事实源

- **禁止**在 `phase2_adapter.py` 或其他地方重复定义 step 列表
- 所有消费方必须调 `drv.get_steps_spec()`
- 锁定测试：`@/UFO/政务平台/phase1_service/tests/unit/test_steps_spec_sot.py:1`（12 个 test）

### 7.7 幂等与重试

- Phase 2 重复调用 `save` 会在服务端产生**二次写入**
- `phase2_idempotency.py` 用 `sha1(case + busi_id + stop_after + name_id)` 做 key，TTL 5 分钟
- **不要**把幂等 TTL 设太长，用户真想重跑就能重跑

### 7.8 字段数（mitm 实录锁定）

- `BasicInfo save` body top-level 必须 **42 个 key**（少一个都不行）
- `MemberPost save` body top-level 必须 **8 个 key**
- 这些数字来自 mitm 抓包实录，变了说明平台升级了字典
- 锁定测试：`test_phase2_bodies.py::TestBasicInfoSaveBody::test_top_keys_count`

## 8. 常见开发任务 How-To

### 8.1 加一个新 API 端点

```
1. 想清楚它属于哪个现有 router（auth/phase1/phase2/matters/system/debug）
   - 如果都不沾边 → 新建 phase1_service/api/routers/<name>.py
2. 在 router 里加 @router.get/post，Pydantic 请求/响应模型放 api/schemas/
3. 业务逻辑放到 phase1_service/api/core/ 新模块（不要在 router 里写逻辑）
4. 如果要调协议 → 从 system/phase2_protocol_driver 的 STEPS_SPEC 里找或加 step
5. 在 api/main.py 里 include_router
6. 在 api/llm_function_schema.py 的对应 group 里加一个 tool 定义
7. 在 phase1_service/cli.py 里加对应子命令
8. 写至少 1 个 pytest 单元测试
9. 跑 python -m pytest 确认绿
```

> **这是"避免做了没转化"的 checklist**。

### 8.2 调试一个协议 bug（比如 step 15 返回 A0002）

```
1. 先看 mitm 抓包里上一次成功的样本
   python -m phase1_service.cli mitm-latest --api-pattern BasicInfo/operationBusinessDataInfo
2. 对比当前 body 和成功 body 的 diff
   - 字段数？ 值类型？ 编码？ 缺字段？
3. 如果是 body 问题 → 改 phase2_bodies.py 对应 builder
4. 如果是 header/cookie 问题 → 改 icpsp_api_client.py 或查 session
5. 如果是 signInfo/aesKey 问题 → 查 phase2_constants.py
6. 改完后跑冒烟：python -m pytest phase1_service/tests/unit/test_phase2_bodies.py
7. 真实验证：
   python -m phase1_service.cli phase2-register --case docs/case_xxx.json --start-from 15 --stop-after 15
```

### 8.3 支持一个新 entType（比如给 1100 有限公司加支持）

```
1. 先跑 Phase 1 拿一个 1100 的 busiId（手工验证可行）
2. 抓包对比 4540 和 1100 的 BasicInfo save body 差异
   → 用 python -m phase1_service.cli mitm-latest --api-pattern BasicInfo
3. 找出差异字段：很可能 orgForm / memberPost 结构不同
4. 修改 phase2_bodies.py 的 builder，按 ent_type 分支
5. 修改 phase2_enums.py 的 helper（如有新枚举）
6. 给新 entType 写一个 case JSON（仿 case_有为风.json）
7. 加一个 test：test_phase2_bodies.py::TestBasicInfoSaveBody::test_for_1100
8. 冒烟：phase2-register --case docs/case_新.json
```

### 8.4 从 mitm 抓包样本提取字段

```
1. 浏览器正常操作一遍，mitmproxy 会把流量存到 dashboard/data/records/mitm_ufo_flows.jsonl
2. 查最新的：
   python -m phase1_service.cli mitm-latest --api-pattern <URL 关键词>
3. 返回完整 req_body + resp_body（明文）
4. 可以导出到 Python dict 直接用 json.loads 对比
```

### 8.5 加一个新 Phase 2 step（比如电子签章 step 26）

```
1. 在 phase2_protocol_driver.py 里加 step26_xxx(client, ctx) 函数
2. 在 STEPS_SPEC 里追加 (26, "name", step26_xxx, False)
3. test_steps_spec_sot.py::test_driver_spec_has_25_steps → 改成 26
4. 在 phase2_bodies.py 加对应 builder（如需）
5. 单元测试：给 builder 加 test_phase2_bodies.py
6. 冒烟：phase2-register --stop-after 26
```

## 11. 已知债务（不影响运行但该知道）

详细清单见 `@/UFO/政务平台/docs/API未覆盖资产盘点_20260424.md:1`。优先级由高到低：

### 架构层
- [ ] **case 仍是弱类型 `Dict[str, Any]`**：缺字段到某 step 才 400 暴露，应改 Pydantic
- [ ] **存储散在 4 处**（token/cookies/latest.json/snapshot）：应抽象 `StateStore` 统一接口
- [ ] **无结构化日志**：FastAPI 默认 access log，故障排查靠 `print`
- [ ] **API 前缀不一致**：`/api/phase1/auth/*` vs `/api/auth/*` 两套风格并存

### 代码层
- [ ] **`system/` 目录杂货铺**：活代码和 106KB 老 CDP 代码混在一起，需拆 core/legacy/tools
- [ ] **长函数**：`phase2_protocol_driver.py` 某些 step 70+ 行，难单测
- [ ] **硬编码路径**：`G:/UFO/政务平台` 到处出现，换机器即废

### 运维层
- [ ] **无 Dockerfile**：无法一键部署
- [ ] **无 `.env` 环境变量**：端口/CDP 地址硬编码
- [ ] **security**：CORS `*`、token 明文存、sysParam snapshot 含密钥但无文件权限

### 业务层
- [ ] **Phase 2 后续电子签章未做**：需 CA 证书集成
- [ ] **仅单样本验证**：跨 entType/跨行业/跨区划的通用性未系统测试
- [ ] **无 Web UI for case 编辑**：用户只能手改 JSON

**之前**：整个流程依赖 CDP（Chrome DevTools Protocol）+ 真实浏览器页面操作，面临三大阻塞：

## 二、核心突破：D0022 越权访问"三层洋葱"

最大技术难点是 `D0022 越权访问`——服务端把 header 不对、body 不对、魔数不对**统统报同一个错误码**，没有任何诊断信息。

### 第 1 层：浏览器指纹头（Header）

服务端严格校验 `sec-ch-ua` / `sec-ch-ua-mobile` / `sec-ch-ua-platform` / `Sec-Fetch-*` 等浏览器自有 header。Python `requests` 默认不带这些。

**解法**：在 `ICPSPClient` 中注入完整的 `_BROWSER_LIKE_HEADERS`（Edge UA + 全套 Sec-Fetch-* 头）。

### 第 2 层：请求 body 结构（Schema）

用 **离线 diff**（`_offline_diff_steps_1_4.py` 等）比对 driver 与 mitm 真实浏览器样本，发现：

| step | 错误做法 | 正确做法 | 后果 |
|------|----------|---------|------|
| step2 | 多发 `extraDto` 整个对象 | 根本不要这字段 | SESSION 进错误分支 |
| step2 | `linkData.continueFlag` | 只要 `{token:""}` | 服务端当"续办" |
| step3 | 漏 `itemId: ""` | 必须带空字符串 | 无法绑定组件实例 |

**解法**：body 以 mitm 备份样本为真理，逐字段对齐。

### 第 3 层：signInfo 魔数（最深一层）

4 条 mitm 首次保存样本，`signInfo` 字段全部是 **`-252238669`**。这不是对 body 做 hash，不是用户相关的 token，就是前端硬编码的一个**已知合法魔数**。动态计算的任何值都被拒。

```python
# system/phase1_protocol_driver.py
ctx.sign_info = -252238669  # 固定魔数，不要动态算
```

### 彩蛋第 4 层：afterNameCheckSign

攻破 D0022 后，step7 返回 `code=00000` 但不给 busiId。mitm diff 发现：二次保存必须带 `afterNameCheckSign: "Y"`（用户已确认近似名）。补上后 → `resultType=0` + busiId 到手。

## 三、方法论凝练

### "样本即真理"四步法

面对黑盒政务服务端，**不猜、不试、只对齐样本**：

```
第 1 步：离线 diff（0 代价，0 风险）
    ↓
第 2 步：分层修正（Header → Schema → Data → Magic）
    ↓
第 3 步：单变量探针（确认 diff 可信后才发请求，步间 ≥15s）
    ↓
第 4 步：浏览器 fetch 确权（同浏览器 + 同用户 + driver body → 区分 header 问题 vs body 问题）
```

### 三原则（可迁移到其他政务/金融系统）

| 原则 | 含义 |
|------|------|
| **样本即合约** | 没有文档的服务端，唯一契约就是"它接受过的样本" |
| **结构先于数据，数据先于魔数** | 按 schema → 值语义 → 具体数值 顺序修 |
| **每次盲请求都是债务** | D0029 冷却 5-10 分钟，离线 diff 让你 0 代价完成 95% 定位 |

### 4.4 核心设计决策

**令牌桶限流**：`AsyncTokenBucket` 内置在 FastAPI 中间件，防止并发调用触发 D0029。

**Authorization 管理**：两种模式——
1. 请求级传入（header 或 body）
2. 从 `runtime_auth_headers.json` 自动读取（浏览器登录后自动同步）

**名称禁限用词检测**：driver_adapter 返回 `step5_result_type` / `step7_result_type`，当名字被禁时 `reason="name_prohibited"` 并附带 `reason_detail`。

## 五、发现的名称禁限用规则

多案件测试中发现服务端的**名称审核机制**：

| resultType | 含义 | busiId | 示例 |
|-----------|------|--------|------|
| **0** | 名称完全通过 | ✅ 分配 | "李陈梦"、"裕鑫" |
| **1** | 含**禁止**使用的内容 | ❌ 拒绝 | "裕安"（安徽六安地名） |
| **2** | 含**限制**使用的内容 | ⚠️ 视情况 | "联创"、"科技" |

**被拒词汇样本**：裕安（地名）、联创（商业词）、永裕科技（限制）、裕鑫电商（限制）  
**通过词汇样本**：李陈梦（人名）、裕鑫（简洁商标词）

> `resultType=2`（限制）+ `afterNameCheckSign="Y"` → 部分可通过，取决于服务端内部评分。

## 七、错误码词典

| 错误码 | 含义 | 应对策略 |
|--------|------|----------|
| `00000` | 成功 | - |
| `GS52010103E0302` | 未认证 / 会话失效 | 重新获取 Authorization |
| `D0022` | 越权访问 | 检查 Header → Body Schema → signInfo 魔数 |
| `D0029` | 操作频繁（限流） | 等待 5-10 分钟冷却 |
| `A0002` | 参数校验失败 | 检查 body 字段编码（如 busiAreaData JSON vs URL 编码） |

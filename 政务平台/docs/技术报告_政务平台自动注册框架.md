
4. **断点续跑透明**：checkpoint 持久化在 Pipeline 层

## 3. 鉴权与会话管理

### 3.1 双 Session 架构（关键）

广西平台特殊点：**同时持有两套 Session**

| Session | 域 | 用途 |
|---------|-----|------|
| `SESSIONFORTYRZ` | `.zwfw.gxzf.gov.cn` | 统一身份认证（SSO 上游） |
| `SESSION` | `zhjg.scjdglj.gxzf.gov.cn` | 业务平台（多个 path 各自独立） |
| `Authorization` (header) | header 32-hex | 业务接口主鉴权 |

**实测：**
- `getUserInfo` 仅靠 `Authorization` 即可
- `loadCurrentLocationInfo`、`name/submit` 等业务接口必须同时有 9087 SESSION cookie
- 两套 cookie 缺一不可，`Authorization` 失效后整套要重新刷

### 3.2 纯 HTTP 扫码登录链（无浏览器）

```
[Step 1] QR 扫码登录
  ├─ GET 登录页 → 拿 csrf_token + SESSIONFORTYRZ
  ├─ POST getQrcode → sessionId + QR 图片 (PNG bytes)
  ├─ 调用 PIL/qrcode 显示二维码 → 用户扫码
  ├─ 轮询 checkScan → 等待 scan_status=success
  └─ POST 提交登录 → SESSIONFORTYRZ 持久化

[Step 2-4] SSO 链 → Authorization
  ├─ Step 2: GET tyrz oauth2 redirect → 6087 SESSION
  ├─ Step 3: GET entservice?sign → ENT_SERVICE session
  └─ Step 4: GET entservice 二次跳转 → Authorization (32 hex)

完成后 ICPSPClient 自动用 SESSION cookies + Authorization
```

**关键代码位置：** `system/login_qrcode_pure_http.py`

```python
def full_login() -> str:        # 全新扫码登录（~30s 含人工扫码）
def refresh_token() -> str:     # 用已存 SESSIONFORTYRZ 静默续期（~2s 无扫码）
def ensure_token() -> str:      # 智能：先验活，失活再 refresh，再不行才扫码
```

### 3.3 Token 续期策略

```
ensure_token()
   │
   ├─ check_token_alive(current) → 活 → return
   │
   ├─ refresh_token() → 用 SESSIONFORTYRZ 重走 SSO 2-4 步
   │     成功 → return new_token
   │     失败（SESSIONFORTYRZ 也过期）↓
   │
   └─ full_login() → 扫码
```

### 3.4 ICPSPClient — HTTP 客户端封装

`system/icpsp_api_client.py`

```python
@dataclass
class ICPSPClient:
    base: str = "https://zhjg.scjdglj.gxzf.gov.cn:9087"

    def post_json(self, path, body, extra_headers=None) -> dict:
        # 自动注入 _headers()：Authorization、Origin、Referer、UA、language
        # 自动加 ?t=毫秒时间戳（模仿前端）
```

## 4. 编排核心：Pipeline 引擎

### 4.1 数据契约（强类型）

`system/orchestrator/core.py`

```python
@dataclass
class StepResult:
    name: str
    ok: bool
    code: str = ""              # 服务端返回 code（如 "00000"）
    result_type: str = ""       # rt（如 "0" / "2"）
    message: str = ""
    extracted: Dict[str, Any]   # 从响应抽取的关键状态
    raw_response: Dict[str, Any]
    duration_s: float = 0.0
    sent_body_keys: List[str]   # 调试用：发送 body 的顶层 keys
    diagnostics: Dict           # SmartDiagnosisHook 写入

@dataclass
class StepSpec:
    name: str
    fn: Callable[[Client, Context], StepResult]
    optional: bool = False      # 失败不阻断
    tag: str = ""
    description: str = ""

@dataclass
class PipelineContext:
    state: Dict[str, Any]       # 跨步骤共享，Hook 可读写
    history: List[StepResult]
    metadata: Dict[str, Any]

class InterventionSignal(Exception):
    """Hook 抛出此异常以请求人工干预（如改名）"""
```

### 4.2 Pipeline 执行流程

```python
class Pipeline:
    def run(self, client, ctx) -> PipelineResult:
        for step in self.steps:
            self._fire("before_step", step, ctx)
            try:
                result = step.fn(client, ctx)
            except InterventionSignal as e:
                self._fire("on_intervention", e, ctx)
                # 由 hook 决定 retry / skip / abort
            self._fire("after_step", step, result, ctx)
            if not result.ok and not step.optional:
                self._fire("on_failure", result, ctx)
                break
        return PipelineResult(...)
```

### 4.3 Hook 生命周期（按 priority 排序）

| Hook | priority | 职责 |
|------|----------|------|
| `LoggingHook` | 10 | 彩色日志输出 |
| `StateExtractorHook` | 30 | 抽取 busiId/nameId/status 到 ctx.state |
| `SmartDiagnosisHook` | 35 | 业务阶段、错误诊断、建议动作 |
| `ThrottleHook` | 40 | 限流/SESSION 过期检测 → InterventionSignal |
| `NameCorrectionHook` | - | 核名 rt=2 时交互改名 |
| `ResultWriterHook` | 90 | Pipeline 结束写入 JSON |

**Hook 可见的钩子：**
```python
class Hook:
    def before_pipeline(self, ctx)
    def before_step(self, step, ctx)
    def after_step(self, step, result, ctx)
    def on_failure(self, result, ctx)
    def on_intervention(self, signal, ctx)
    def after_pipeline(self, result, ctx)
    def on_step_success(self, step, result, ctx)  # 用于断点持久化
```

### 4.4 Adapter 模式 — 把驱动函数适配进 Pipeline

`system/orchestrator/adapters.py`

```python
class Phase1Adapter:
    def __init__(self, case):
        self.driver_ctx = DriverContext.from_case(case)

    def _wrap(self, fn):
        """把 (client, DriverCtx) -> (P1StepResult, resp)
            包装为 (client, PipelineCtx) -> StepResult"""
        def adapted(client, pipeline_ctx):
            p1_result, resp = fn(client, self.driver_ctx)
            return StepResult(
                name=p1_result.name,
                ok=p1_result.ok,
                code=str(p1_result.code),
                ...
                raw_response=resp,
            )
        return adapted

    def make_steps(self) -> List[StepSpec]:
        return [
            StepSpec("checkEstablishName", self._wrap(check_establish_name), optional=True),
            StepSpec("loadCurrentLocationInfo", self._wrap(p1_load_loc), optional=True),
            ... # 8 步
        ]
```

**核心收益：**
- Pipeline 完全不知道 DriverContext 的存在
- 改名重建 DriverContext 时 `adapter.rebuild(new_case)` 即可，不丢 adapter 实例

### 5.3 关键发现 — `signInfo` 是固定魔数

> 服务端把 signInfo 当作"已知合法魔数"校验，**与 body 内容、用户都无关**。
> 用动态 Java hashCode 任何变化都触发 D0022 越权访问。
> mitm 抓包 4/4 首次保存样本的 signInfo 全是 `-252238669`。
> **直接锁定为该固定值。**

### 5.4 改名流程（Hook 干预）

```
   ├─ 提示用户改字号
   ├─ adapter.rebuild(new_case)
```

### 6.6 动态 signInfo 追踪

不同于 Phase 1 的固定魔数，Phase 2 各组件 save 需要服务端 load 时返回的实时 signInfo。

```python
def _preload_component_sign_info(client, c, comp_url):
    """save 前先 load 拿最新 signInfo（★ 浏览器行为）"""
    load_resp = client.post_json(establish_comp_load(comp_url), load_body)
    si = (load_resp.data.busiData).get("signInfo")
    c.snapshot["last_sign_info"] = si
    c.snapshot[f"{comp_url}_signInfo"] = si

def _post_save_and_track_sign(client, url, body, c):
    body["signInfo"] = c.snapshot["last_sign_info"]
    resp = client.post_json(url, body)
    # save 响应可能更新 signInfo，下次 save 用新值
    new_si = resp.data.busiData.get("signInfo")
    if new_si:
        c.snapshot["last_sign_info"] = new_si
    return resp
```

### 6.7 linkData 的 `busiCompComb` 之谜

服务端 load 返回的 `linkData` 包含一个庞大的 `busiCompComb` 对象（设立流程的全部组件配置）：

```json
"busiCompComb": {
  "id": "22561619020183",
  "compComb": "NameInfoDisplay,BasicInfo,MemberPost,...32 个组件",
  "compCombName": "[NameInfoDisplay]名称信息,...",
  "processToBusiComp": "nameshow|0,fill|1-26,confirm|27,..."
}
```

**实测：**
- `save` 请求体里**不能传** `busiCompComb`（mitm 抓真实浏览器请求验证）
- `producePdf` 请求体里**也不需要** `busiCompComb`
- 各 load 之间用 `prev_ld` 携带 `busiCompComb` 是为了 server-side 路径校验

### 6.8 `signInfo=ESTABLISH 魔数`

Phase 2 establish 的 SIGN_INFO_ESTABLISH 也是固定魔数（不是动态 hashCode）。值在 `phase2_bodies.py` 顶部常量。

### 6.9 producePdf A0002 的真相

> **这是服务端临时异常**，与请求体构造无关。
> 真实浏览器（Edge + mitm 抓包）同样得到 `code=A0002 / msg="服务端异常"`。
> 浏览器顶部红色提示："由于系统服务出现异常，请勿多次重复提交，耐心等待片刻再重新尝试"
> **它正好作为"云提交"的天然停点，不影响测试目的**。

```python
TARGETS = ["producePdf", "YbbSelect/operationBusinessDataInfo", ...]

        if any(t in flow.request.url for t in TARGETS):
```

## 13. 已知 / 已解决的"坑"

| 坑 | 现象 | 真因 | 解决 |
|----|------|------|------|
| 动态 hashCode signInfo | D0022 越权 | 服务端校验固定魔数 | 锁定 `-252238669` |
| Cookie 与 header 双写 | 部分接口 401 | requests 与 header Cookie 冲突 | header 移除 Cookie，让 session 自管 |
| visaFree 参数缺失 | SPA 不初始化 | 前端路由守卫 | URL 加 `&visaFree=true` |
| MemberPost 直接 save A0002 | NPE | 服务端没 member 记录 | 先 MemberBaseInfo save 再 list 取 itemId |
| 续跑漂移到错位置 | StepNotAdvanced | 本地 checkpoint 与服务端不一致 | StateExtractorHook 用 server probe 覆盖 |
| 同账号被浏览器顶号 | 全部 GS52010103E0302 | 单点登录踢号 | 抓包后必 refresh_token |
| producePdf A0002 | 服务端异常 | 服务端临时故障（与请求无关） | 在 YbbSelect 停下作为天然停点 |

## 15. 总结

这套框架的核心价值：

1. **协议优先**：99% 的流程通过纯 HTTP 协议完成，没有浏览器、CDP、Puppeteer 的依赖
2. **数据驱动**：换公司只需改一个 JSON case 文件
3. **可恢复**：任何中断都能从 checkpoint 续跑，且自动用服务端探针校正
4. **可扩展**：Pipeline + Hook + Adapter 三件套让加新组件、新省份、新业务类型都不破坏现有代码
5. **可观测**：每一步的 sent_body / response / extracted / diagnostics 全持久化到 records，便于事后分析

> 唯一需人介入的环节：**初次扫码登录**（30s）和**核名 rt=2 时改字号**（NameCorrectionHook 主动发起交互）。
> 其余全程无人值守。

**当前已验证场景：广西容县 · entType=4540 个人独资 · busiType=02 设立登记 · 走到 YbbSelect 云提交停点**。

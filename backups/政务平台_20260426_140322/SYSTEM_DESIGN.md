# 政务助手系统 — 合规架构设计

## 核心理念

**合规第一，效率第二。认证环节走原系统原流程，机器只做填表和跟踪。**

```
客户 → 我们的系统(LLM审核+代填) → 政务平台(原系统认证) → 结果跟踪
                ↓                          ↓
          材料审核/预填              人脸/短信/银行卡认证
          问题反馈/补正              必须客户本人完成
```

## 合规红线

| 红线 | 说明 | 我们的策略 |
|---|---|---|
| **不绕过认证** | 人脸、短信、银行卡认证必须本人完成 | 暂停流程 → 通知客户 → 客户在原系统完成 → 继续 |
| **不伪造身份** | 不能用他人身份提交 | 客户自己注册账号，我们只代操作已授权账号 |
| **不篡改请求** | 不能修改认证相关的请求/响应 | 代理网关对认证接口只透传，不拦截不修改 |
| **不绕过审核** | 政务平台的审核流程必须走完 | 我们只帮填表，审核结果原样返回 |
| **材料真实性** | 提交的材料必须真实 | LLM 审核材料真实性，有问题拒绝提交 |

## 业务流程

### 流程一：新客户注册+设立登记

```
1. 客户提交材料 → 我们系统
2. LLM 审核材料
   ├─ 不合格 → 反馈客户补正 → 回到1
   └─ 合格 → 继续
3. 系统自动填写政务平台表单（CDP/API）
4. 提交前 → 生成预览 → 客户确认
5. 客户确认 → 系统提交
6. 政务平台要求认证？
   ├─ 是 → 暂停 → 通知客户"请完成认证"
   │       → 客户打开政务平台完成人脸/短信认证
   │       → 系统检测认证完成 → 继续
   └─ 否 → 继续
7. 系统跟踪办件进度
   ├─ 审核通过 → 通知客户"已通过"
   ├─ 需要补正 → LLM分析补正要求 → 通知客户 → 回到1
   └─ 被驳回 → 通知客户 + 原因分析
```

### 流程二：变更登记

```
1. 客户提交变更需求+材料
2. LLM 审核 → 识别变更类型 → 匹配政务平台路由
3. 自动填写变更表单
4. 客户确认 → 提交
5. 跟踪进度 → 通知结果
```

### 流程三：进度查询（全自动）

```
1. 系统定时轮询办件进度
2. 状态变化 → 通知客户
3. 需要操作 → 触发对应流程
```

## LLM 在系统中的角色

```
┌─────────────────────────────────────┐
│            LLM 引擎                  │
│                                      │
│  1. 材料审核                         │
│     • 检查材料完整性                  │
│     • 检查信息一致性                  │
│     • 检查格式合规性                  │
│     • 生成补正建议                    │
│                                      │
│  2. 表单映射                         │
│     • 客户材料 → 政务平台字段映射      │
│     • 识别业务类型 → 匹配路由          │
│     • 生成填写方案                    │
│                                      │
│  3. 结果分析                         │
│     • 解读政务平台审核意见             │
│     • 生成补正方案                     │
│     • 生成进度报告                    │
│                                      │
│  4. 客户沟通                         │
│     • 生成通知消息                    │
│     • 回答客户问题                    │
│     • 解释流程要求                    │
│                                      │
│  ⛔ LLM 不能做                       │
│     • 不能代替客户做认证              │
│     • 不能修改认证数据                │
│     • 不能绕过任何合规检查            │
└─────────────────────────────────────┘
```

## 认证环节的合规处理

### 场景：政务平台弹出人脸认证

```
系统检测到认证要求
    ↓
暂停当前任务，状态标记为 "等待客户认证"
    ↓
通知引擎 → 给客户发消息：
  "您的设立登记已提交，政务平台要求完成人脸认证。
   请打开以下链接完成认证：
   https://zhjg.scjdglj.gxzf.gov.cn:9087/icpsp-web-pc/portal.html#/...
   认证完成后系统将自动继续。"
    ↓
CDP 浏览器保持页面等待（不关闭，不操作）
    ↓
CDP 轮询检测认证状态变化
    ↓
认证完成 → 系统继续提交流程
    ↓
通知客户"认证已完成，流程继续"
```

### 场景：短信验证码

```
系统填写表单到需要验证码的步骤
    ↓
暂停，通知客户：
  "请提供收到的短信验证码（发送到尾号xxxx的手机）"
    ↓
客户回复验证码
    ↓
系统填入验证码 → 继续
    ↓
（验证码必须客户本人手机收到，我们不截取不伪造）
```

## 技术实现

### 目录结构（新增）

```
政务平台/
├── system/                    # 服务系统核心
│   ├── core/
│   │   ├── engine.py          # 主引擎：任务调度、流程控制
│   │   ├── llm_client.py      # LLM 接口（材料审核、表单映射）
│   │   ├── task_manager.py    # 任务管理（状态机）
│   │   └── notifier.py        # 通知引擎（微信/短信/邮件）
│   ├── flows/                 # 业务流程
│   │   ├── establish.py       # 设立登记流程
│   │   ├── change.py          # 变更登记流程
│   │   ├── cancel.py          # 注销登记流程
│   │   └── track.py           # 进度跟踪流程
│   ├── compliance/            # 合规控制
│   │   ├── auth_guard.py      # 认证守卫（检测+暂停+通知）
│   │   ├── proxy_rules.py     # 代理规则（哪些可代理，哪些只透传）
│   │   └── audit_log.py       # 审计日志
│   └── api/                   # 客户接口
│       ├── client_api.py      # 客户 REST API
│       └── webhook.py         # 回调接口
└── ...
```

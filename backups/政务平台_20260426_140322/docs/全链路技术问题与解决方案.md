# A. 浏览器自动登录（账号密码 + 滑块）

## A1. CDPSession.send() 吞掉 Fetch 事件

**现象**：启用 `Fetch.enable` 后，`Fetch.requestPaused` 事件从未被捕获，ssc 拦截始终失败，浏览器永远卡在社保卡扫码页。

**排查过程**：
1. 确认 Fetch.enable 返回成功
2. 手动在 Chrome DevTools 里能看到 ssc 请求
3. 对比 `_login_fetch_v2.py`（独立验证脚本）发现它能成功
4. 定位到 CDPSession 封装层

**根因**：`CDPSession.send()` 内部有 `while True: recv()` 循环，只匹配 `response.id == request.id`，**所有其他 CDP 事件被静默丢弃**：
```python
def send(self, method, params=None):
    self._id += 1
    self.ws.send(...)
    while True:
        resp = json.loads(self.ws.recv())
        if resp.get("id") == self._id:  # ← 只看这个
            return resp.get("result", {})
        # ← Fetch.requestPaused 在这里被无声丢掉！
```

**解决**：Fetch 阶段**关闭 CDPSession，新建 raw WebSocket**。自定义 `_raw_send()` 内联处理 Fetch 事件：
```python
def _raw_send(method, params=None):
    while True:
        msg = json.loads(_ws.recv())
        if msg.get("method") == "Fetch.requestPaused":
            handle_fetch(msg)  # 内联处理
            continue
        if msg.get("id") == mid:
            return msg.get("result", {})
```

**教训**：封装库不透明时，**必须读源码确认消息分发逻辑**。

## A2. 直接构造 tyrz URL 导致 entservice 失败

**现象**：手动拼接 tyrz 登录 URL 并成功登录后，调用 `9087/entservice` 被重定向回 tyrz 登录页。

**根因**：9087 服务端在 SSO 发起时（通过 enterprise-zone 入口）会记录 SSO 上下文。跳过 9087 直接去 tyrz，9087 没有上下文，后续 entservice 拒绝。

**解决**：**必须从 9087 enterprise-zone 入口发起 SSO 链**：
```
✗ 直接去 tyrz → 登录 → entservice 失败（无上下文）
✓ enterprise-zone → 自动 302 到 tyrz → 登录 → entservice 成功
```

**教训**：SSO 链不能跳步，每一跳都可能在服务端设置上下文状态。

## A3. 旧 localStorage 导致 SPA 不触发 SSO 重定向

**现象**：清除 cookies 后导航到 enterprise-zone，9087 SPA 停留在 `/login/404` 或 `/index/enterprise`，不跳转到 tyrz。

**根因**：SPA 从 `localStorage.Authorization` 读取 token。token 存在就认为已登录，不触发 SSO 重定向。**单清 cookies 不够**。

**解决**：使用 CDP `Storage.clearDataForOrigin` 清除 localStorage，**无需导航到对应域**：
```python
cdp.send("Storage.clearDataForOrigin", {
    "origin": "https://zhjg.scjdglj.gxzf.gov.cn:9087",
    "storageTypes": "local_storage,session_storage",
})
```

## A4. ssc.mohrss.gov.cn 阻塞登录流程

**现象**：tyrz 登录成功后，浏览器跳转到 `ssc.mohrss.gov.cn`（人社部社保卡扫码页），需要扫实体社保卡，完全无法自动化。

**关键发现**：6087 的 SESSION cookie 在跳转到 ssc **之前**就已经设置好了。ssc 只是二次验证，跳过不影响 session 有效性。

**解决**：CDP Fetch 拦截 ssc 请求，伪造 302 重定向到 6087 portal：
```python
Fetch.fulfillRequest({
    requestId: rid,
    responseCode: 302,
    responseHeaders: [{"name": "Location", "value": "6087 portal URL"}],
    body: ""
})
```

## A5. Fetch.disable 后残余 paused 请求导致浏览器卡死

**现象**：拦截 ssc 后 break 出循环，浏览器页面无法继续加载，白屏卡死。

**根因**：break 后 Fetch 仍处于 enable 状态，6087 portal 加载时发出的子资源请求也被 pause，但未被 continue 或 fulfill，浏览器挂起等待。

**解决**：`_raw_send("Fetch.disable")` 时，内联处理所有残余 `Fetch.requestPaused` 事件（对非 ssc 请求 `continueRequest`），直到收到 `Fetch.disable` 的 response：
```python
while True:
    msg = json.loads(ws.recv())
    if msg.get("method") == "Fetch.requestPaused":
        ws.send(json.dumps({"method": "Fetch.continueRequest", "params": {"requestId": rid}}))
        continue
    if msg.get("id") == disable_msg_id:
        break  # disable 完成
```

## A6. tyrz 滑块 CDP Input 事件无效

**现象**：使用 CDP `Input.dispatchMouseEvent` 拖拽滑块，滑块纹丝不动。

**根因**：tyrz 的滑块库（vue-puzzle-vcode 变体）只监听 DOM `MouseEvent`，**不响应 CDP 合成的 Input 事件**。这是一种反自动化手段。

**解决**：通过 JS `dispatchEvent(new MouseEvent(...))` 在 DOM 层面模拟：
```javascript
// mousedown → 滑块元素
moveBlock.dispatchEvent(new MouseEvent('mousedown', {bubbles:true, clientX:x, clientY:y}));
// mousemove → document（滑块库全局监听）
document.dispatchEvent(new MouseEvent('mousemove', {clientX: x+distance, clientY: y}));
// mouseup → document
document.dispatchEvent(new MouseEvent('mouseup', {...}));
```

**教训**：遇到 CDP Input 无效时，先检查目标是否使用了 DOM 事件监听而非 CDP 事件通道。

## A7. 滑块图片面板默认隐藏

**现象**：读取背景图 `img.backImg` 的 `naturalWidth` 为 0，OpenCV 匹配无法工作。

**根因**：tyrz 滑块将图片容器 `.verify-img-out` 设为 `display:none`，鼠标 hover 时才显示。CDP 自动化无法触发原生 hover 态。

**解决**：JS 强制设置 display 后再读取图片：
```javascript
imgOut.style.display = 'block';
imgOut.style.position = 'absolute';
imgOut.style.zIndex = '99999';
```

## A8. 滑块连续失败后图片返回空

**现象**：`bg=0 bytes`，OpenCV 解码失败。

**根因**：tyrz 服务端限流，连续失败多次后图片接口返回空数据。

**解决**：等待 30-60 秒后重试。脚本内置 5 次尝试 × 指数退避。

# B. 扫码登录

## B1. tyrz `/sso/oauth2/authorize` 返回 nginx 404

**现象**：submitLogin 返回的 redirect URL `https://tyrz.zwfw.gxzf.gov.cn/sso/oauth2/authorize?...`，无论 Python requests、浏览器 fetch()、带代理 8080，全部返回 **nginx 404**。

**排查过程**（历时最长的问题）：
1. 确认 DNS 解析正常（113.12.121.4）
2. 检查 hosts 文件 — 无特殊配置
3. 添加 Referer / User-Agent / sec-ch-ua — 无效
4. 代理 8080 — 同样 404（Content-Length 不同：153 vs 555，但都是 nginx 404）
5. 浏览器内 XHR — 同样 404
6. 浏览器页面导航 — 也显示 404 页面

**根因**：tyrz 的 nginx **只配了 `/am/` location 对外暴露**。`/sso/oauth2/authorize` 是内部路径，只有 **server-side redirect**（由内部 upstream 处理）时才可达。从外部任何方式直接访问都是 404。

**解决**：放弃纯 HTTP 走 OAuth2 redirect 链。改用浏览器 CDP 从 **enterprise-zone 入口**导航，让 SSO 的 server-side redirect 链自然完成。浏览器页面导航不受 `/sso/` 限制（302 在服务端内部跟随）。

**最终架构**：
```
Steps 1-5 纯 HTTP: tyrz → QR 码 → 扫码 → submitLogin → SESSIONFORTYRZ
Step 6 浏览器 CDP: 注入 cookie → enterprise-zone → SSO 链 → token
```

**教训**：SSO 内部路径不等于公网可达路径。不要假设 redirect URL 可以直接访问。

## B2. HTTP 请求 tyrz 超时卡死

**现象**：Python requests 请求 `tyrz.zwfw.gxzf.gov.cn` 一直卡住不返回，脚本挂起。

**根因**：系统配置了代理（Fiddler/mitmproxy），requests 默认读取环境变量走代理。代理挂了或慢了导致超时。

**解决**：
```python
s.proxies = {"https": None, "http": None}  # 显式绕过代理
s.timeout = 10  # 必须设超时
```

**教训**：所有网络请求**必须同时设置 proxies=None 和 timeout**。

## B3. 6087 authLogin 返回 JSON 而非 302

**现象**：`GET 6087/TopIP/sso/authLogin?authType=zwfw_guangxi` 返回：
```json
{"msg":"没有权限","code":"-8","data":{"redirect_uri":"..."}}
```
期望是 302 redirect。

**根因**：requests Session 的默认 `Accept` 头是 `application/json`。6087 服务器**根据 Accept 决定响应格式**：
- `Accept: application/json` → 返回 JSON 错误体
- `Accept: text/html` → 返回 302 redirect

**解决**：
```python
headers = {"Accept": "text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8"}
```

**教训**：同一个接口对不同 `Accept` 头返回完全不同的行为。调试时先确认请求头是否符合预期。

## B4. Fetch 事件堆积导致浏览器挂起

**现象**：导航后用 `time.sleep(15)` 等待 SSO 完成，浏览器页面白屏卡死。

**根因**：sleep 期间 `Fetch.requestPaused` 事件在 WebSocket 缓冲区堆积，无人读取。浏览器因为请求被 pause 但未 continue 而挂起。

**解决**：用主动消息循环替代 sleep：
```python
ws.settimeout(3)
while time.time() < deadline:
    try:
        msg = json.loads(ws.recv())
    except:
        continue  # 超时 → 检查页面状态
    if msg.get("method") == "Fetch.requestPaused":
        handle_fetch_event(msg)
```

**教训**：**CDP Fetch 启用期间，绝对不能 sleep**。必须持续读取 WebSocket 消息。

## B5. 重复登录触发风控

**现象**：每次调试路由就重新扫码登录，试了十几次后 QR 码接口响应变慢。

**根因**：tyrz 有风控机制，短时间内同一账号多次登录会被限流。

**解决**：
- **登录只做一次**，后续用已建立的 session 探测路由
- 保持类人节奏（导航间 2-5 秒延时）
- 先分析（读文档、看 JS 源码），再行动

**教训**：实网调试的黄金法则 — **每一次请求都有代价，先想清楚再发**。

## B6. checkQrCode 参数名不是 sessionId

**现象**：POST 参数 `sessionId=xxx` 无效，轮询永远返回"未扫码"。

# C. 纯协议名称登记（Browser Distillation）

## C1. D0022 越权访问 — 浏览器指纹头

**现象**：纯 Python requests 调用 9087 API，所有接口统一返回 `D0022 越权访问`，没有任何具体错误信息。

**根因**：服务端严格校验浏览器指纹 header：`sec-ch-ua` / `sec-ch-ua-mobile` / `sec-ch-ua-platform` / `Sec-Fetch-Dest` / `Sec-Fetch-Mode` / `Sec-Fetch-Site`。Python requests 默认不带这些。

**解决**：从浏览器抓包复制完整 header 集：
```python
headers = {
    "sec-ch-ua": '"Microsoft Edge";v="131", "Chromium";v="131"',
    "sec-ch-ua-mobile": "?0",
    "sec-ch-ua-platform": '"Windows"',
    "Sec-Fetch-Dest": "empty",
    "Sec-Fetch-Mode": "cors",
    "Sec-Fetch-Site": "same-origin",
    # + Authorization, Content-Type, Origin, Referer ...
}
```

---

## C2. D0022 越权访问 — 请求 body 结构

**现象**：加了完整 header 后，部分接口仍返回 D0022。

**根因**：请求 body 多了或少了字段，导致服务端路由到错误分支。例如：
- `loadBusinessDataInfo` 必须含 `itemId: ""`（空字符串，不是省略）
- `operationBusinessDataInfo` 的嵌套 JSON 结构必须与浏览器 byte-exact 一致

**解决方法论 — "样本即真理"四步法**：
1. **离线 diff**：把浏览器真实请求（mitm 抓包）与 Python 构造的请求做**递归字段级 diff**
2. **逐字段修正**：每一条差异都是嫌疑
3. **单字段验证**：一次只改一个字段，确认效果
4. **不猜不试**：不发请求就解决 90% 的问题

---

## C3. D0022 越权访问 — signInfo 魔数

**现象**：header 和 body 都对齐后，`operationBusinessDataInfo` 仍返回 D0022。

**根因**：请求体中有一个 `signInfo` 字段，值是 `-252238669`。这是服务端用 Java `String.hashCode()` 算法对特定字符串的签名。**不是随机数，是固定魔数**。

**排查过程**：
1. 逆向前端 JS 找到 signInfo 计算逻辑
2. 发现是 Java hashCode 算法
3. Python 实现同样算法得到一致值

**解决**：硬编码魔数（经实测服务端不严格校验变化的 signInfo，用样本值透传即可）。

**教训**：D0022 是一个**万能垃圾桶错误码**。header 不对、body 不对、魔数不对都返回同一个。**只能逐层排除**。

# D. 实网调试与运维

## D1. 旧草稿上下文污染新建流程

**现象**：带 `entType=1100` 入口参数启动，页面仍回到旧"个人独资"名称流程，显示旧草稿名称。

**根因**：SPA localStorage 中 `_topnet_flowParams` 含旧草稿上下文，`flow-control` 初始化优先读取已有续办上下文。

**解决**：
1. 清理 `_topnet_flowParams` / `continueFlag`
2. 调 `flow-control.initData()` 重建无 `busiId` 的新建态
3. 启动前做"上下文体检"，发现旧 `busiId` 先清理

## D2. 9087 后端间歇性 500

**现象**：所有 9087 API（包括 `actuator/health`）返回 500，token 有效但业务不可用。

**排查**：
- 静态资源（SPA HTML）正常 200
- nginx 转发正常
- Spring Boot 错误格式（`{"timestamp":..., "status":500}`）
- 无 auth 访问 `/icpsp-api/` 返回"认证失败"（说明后端进程在但不健康）

**根因**：政务平台夜间维护（晚 11 点左右），后端服务重启或数据库维护。

**解决**：等待恢复。脚本加 `--wait-server` 参数自动轮询等待：
```python
for i in range(30):  # 最多等 15 分钟
    time.sleep(30)
    if check_server_health():
        break
```

**教训**：政务系统有维护窗口，**调试安排在工作时间**成功率更高。

# 附录：关键方法论

## 方法论 1：样本即真理

面对"服务端黑盒 + 万能错误码"的场景：
1. **抓浏览器真实请求**（mitm/Fiddler）作为"真理源"
2. **离线 diff**（不发请求）找出差异
3. **逐字段修正**，一次改一个
4. **绝不猜测**

## 方法论 2：单步推进

实网调试纪律：
1. 执行一步
2. 等 2-5 秒（类人延时）
3. 读取反馈（HTTP 响应 / 页面状态）
4. 确认成功后再下一步
5. 失败立即停止，分析后再试

## 方法论 3：Browser Distillation（浏览器蒸馏）

```
浏览器的唯一作用 = 获取 Authorization token（32-hex）
全部业务逻辑 = Python 纯 HTTP 请求
```

好处：
- 不受 SPA 渲染延迟影响
- 不受 CSS/JS 加载阻塞影响
- 速度从分钟级降到秒级
- 可批量、可自动化

# E. 终极突破：纯 HTTP 全流程（2026-04-22 深夜）

## E1. 发现 /am/oauth2/authorize 可达

**背景**：submitLogin 返回的 redirect URL 走 `/sso/oauth2/authorize`（404），之前只能用浏览器 CDP 完成 SSO 链。

**发现**：正常 SSO 入口（从 enterprise-zone 触发）走 `/am/oauth2/authorize`，**不是** `/sso/`。带 SESSIONFORTYRZ 访问 `/am/` 路径，tyrz 自动授权。

**方法**：不走 submitLogin 的 redirect URL，走**正常 SSO 入口**（entservice → 6087/authLogin → tyrz /am/ → 6087 code）。

---

## E2. Bridge "加载中"页面的秘密

**现象**：6087/sso/oauth2 和 9087/entservice?sign 都返回同一个 "加载中" HTML 页面。

**发现**：这是一个 JS bridge 页面，行为如下：
- GET 请求 → nginx 返回 bridge HTML（静态）
- POST 请求（带 `X-Requested-With: XMLHttpRequest`） → nginx 转发到后端，返回 JSON
- `code=0` → 登录成功，redirectUrl 指向下一步
- `code=-9` → 需要 redirect 到 tyrz 授权

**解决**：Python 模拟 bridge JS 行为：POST 同一 URL，带 XHR header，解析 JSON 响应。

---

## E3. 两轮 entservice?sign 才能拿到 Authorization

**这是最关键的发现。**

**现象**：第一轮 entservice?sign bridge POST 返回 code=0，但 redirectUrl="/TopIP/"（无 Authorization）。

**排查**：反复测试发现 entservice 行为与 session 状态有关：
- 无 ENT_SERVICE session → sign bridge 只建立 session，redirectUrl="/TopIP/"
- 有 ENT_SERVICE session → sign bridge 返回完整 redirectUrl，含 `Authorization=32位hex`

**根因**：9087 entservice 的两阶段设计：
1. 第一次 sign → 建立 9087 session（`lastAuthType=ENT_SERVICE`），但不生成 Authorization
2. 第二次 sign → 检测到已有 ENT_SERVICE session → 生成 Authorization 并放入 redirectUrl

**最终流程（4 步纯 HTTP）**：
```
Step 1: QR 扫码 → SESSIONFORTYRZ
Step 2: entservice → 6087 bridge (code=-9 → tyrz → code=0) → 6087 SESSION
Step 3: 清 /icpsp-api → entservice → sign bridge → ENT_SERVICE session（无 Authorization）
Step 4: 保留 session → entservice → sign bridge → redirectUrl 含 Authorization ✓
```

**结果**：Authorization 在 URL hash fragment 中：
```
portal.html#/login/authPage?Authorization=b00a0a4da756462faa89c3e69be167ad
```

**意义**：**整个登录+注册流程完全不需要浏览器**。从 QR 扫码到获取 token 到 Phase1 名称登记，全部纯 HTTP。

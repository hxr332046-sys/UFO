# 逆向复现方法论：上下文先行、框架撑骨、数据后填

> **地位**：与 `LLM政务自动化框架_目标与实施标准.md` 互补——该文件偏产品与工程标准；本文偏 **复现政务类 SPA 时的思维顺序**，供逆开发、脚本与后续 LLM 规划共用同一套语言。  
> **维护**：方法论不变可少改；若仓库内「状态模型 / 观测链」有重大升级，应同步修订本文「与仓库的对应」一节。

---

## 1. 核心命题

**高水准的逆向复现，不是「先背接口与字段」，而是先回答：系统在何种上下文里、按何种顺序接受合法输入。**

若跳过上下文与流程，直接堆数据或重放抓包，常见后果是：

- 同一请求在「未登录 / 未选区划 / 未关闭 MessageBox」下行为完全不同；  
- 接口 `code=00000` 与页面仍报错并存，无法定位真因；  
- 自动化看似在跑，实为 **空转或走错分支**。

因此采用 **三层顺序**（下文展开）：

1. **上下文与状态对齐**（原系统「在什么处境下」才认你的动作）；  
2. **流程与逻辑的骨架**（用确定性框架把「观测 → 动作 → 再观测」撑起来）；  
3. **再按业务先后填入数据**（名称、住所、材料路径等）。

这与「先普查、再 LLM」「先框架、后填数」的工程决策一致。

---

## 2. 第一层：上下文管理（与原版本「一样」的抽象）

原系统通常是 **SPA + Token + 路由 + 办件实例**，状态一变，同一 URL 或同一接口语义可能完全不同。

复现侧需要显式管理（不必一开始就做满，但要 **有意识建模**）：

| 维度 | 含义（示例） |
|------|----------------|
| **会话** | 是否已登录、`Authorization` / 顶栏是否办件中心 |
| **业务实例** | 当前办件、草稿、`busiType` / `entType` |
| **页面阶段** | `hash`、`#/flow/base/...`、名称子应用 `guide/base` 等 |
| **门控** | 弹窗、级联是否完成、实人/短信是否出现 |

**没有这层对齐**，所谓「复现」只是 **HTTP 层面的偶然成功**，不可迁移、不可审计。

**与仓库的对应**：`l3_step_code`、`packet_chain` 的 `steps` / `blocker_evidence`、`ufo.gov_task_run.v1` 的 `task.events`、逆开发字典 V2 等，都是在把「上下文」从脑子里搬到 JSON。

---

## 3. 第二层：框架撑骨（流程与逻辑，先于填数）

在上下文可被观测的前提下，用 **确定性** 结构把流程撑起来：

- **观测**：DOM / 文案 / 可选 Network hook，形成「当前在哪一步」的证据；  
- **合法动作白名单**：导航、点主按钮、关弹窗、级联、上传（每槽间隔等——见类人节奏规则）；  
- **闸门**：云提交、实人不自动点；失败可分类、可记录。

**LLM 若参与**，只应生成 **结构化计划**，经校验后调用上述工具，而不是替代「骨架」去乱点。

**与仓库的对应**：`packet_chain_portal_from_start`、`cdp_attachment_upload`、`human_pacing`、`acceptance` / 验收线脚本等——**先能稳定「走到哪、卡在哪」**，再谈自动填表。

---

## 4. 第三层：按正确逻辑填数据

只有在 **（1）上下文成立、（2）当前步骤允许输入** 时，才填入：

- 主体名称、住所、资本、人员、材料路径等；  
- 顺序须符合 **页面与校验链**（先名称再住所、先关弹窗再下一步等），而非资料文档里的书写顺序机械照搬。

填数层常见形态：

- **人工**：最稳，适合首通与取证；  
- **脚本**：按 label / Vue 路径注入，与 `case JSON` / `material_pack` 对齐；  
- **LLM**：在固定 Observation 与工具集下补全「下一步填什么」，仍受骨架约束。

**与仓库的对应**：`docs/case_*.json`、`config/material_pack.schema.json`、`rehearsal_assets` 等是「数据层」载体；**普查与 `census_run_record`** 记录「填错顺序时系统如何报错」。

---

## 5. 与「抓包重放」的关系

- **抓包**解决的是：**协议长什么样、头与体怎么组**。  
- **上下文 + 框架**解决的是：**在什么时刻、对谁发才合法**。  

二者缺一不可：只有抓包，容易变成 **静态重放**；只有框架没有协议，容易 **瞎点**。天才级复现通常 **先能说明上下文与步骤图，再让抓包服务于每一步的断言与回归**（如重放断言、`code` 与 UI 对账）。

---

## 6. 小结（一句话）

**先对齐与原系统同构的上下文与流程骨架，再按业务先后与闸门规则往里填数据**——这是政务类 SPA 逆向复现的可扩展做法，也是本仓库自动化与文档多条原则的共同底层思想。

# 两阶段业务流程与开发框架编排

> 目标：把**业务页面流程**、**协议步骤**、**代码框架**三者统一到同一张图里，后续开发、续跑、调试都按这个口径执行。
> 
> 当前主线以 **4540 个人独资** 为主；`1151` 有限公司在第二阶段会多出若干页面，但总编排原则一致。

---

## 1. 业务总览

整个业务应只按 **两阶段** 理解：

### 第一阶段：名称信息（核名）

- 左侧大步骤：`1 名称信息`
- 目标：确认拟用名称可用，并完成名称提交
- 阶段产物：`busiId`、`nameId`、名称阶段完成状态

### 第二阶段：信息填报

- 左侧大步骤：`2 信息填报`
- 目标：完成设立登记表单、材料上传、领照方式、云帮办模式选择
- 阶段终点：到达 **云提交就绪页**，即 `PreSubmitSuccess`

### 自动化边界

- **允许**：推进到云提交就绪、成功页加载完成
- **不允许**：点击最终提交、把业务真正送入云处理/签章/审批链

---

## 2. 第一阶段：名称信息（核名）

## 2.1 UI 视角

从你提供的页面看，第一阶段页面核心字段包括：

- 企业名称
- 企业类型（中类）
- 经办人
- 登记机关
- 名称保留截止

这一步在业务上叫“核名”，但对系统来说，不只是一个查重动作，而是一整段**名称阶段子流程**。

## 2.2 业务目标

- 校验名称是否可用
- 补充名称阶段所需的辅助信息
- 填写投资人/股东摘要信息
- 完成名称提交
- 形成后续设立阶段可继续办理的办件上下文

## 2.3 推荐业务子流程

按业务理解，第一阶段应整理为下面 7 个子动作：

1. 名称预检 / 查重
2. 名称页面初始化
3. 名称补充信息填写
4. 投资人 / 股东摘要填写
5. 保存并进行二次查重确认
6. 名称提交
7. 名称成功页确认

## 2.4 当前协议映射

当前代码里，这一阶段实际上被拆散在两个驱动器中：

| 业务子流程 | 当前代码位置 | 说明 |
|------|------|------|
| 名称预检 / 初始化 / 禁限词 | `system/phase1_protocol_driver.py` | 当前的纯协议 Phase 1 主驱动 |
| 首次保存 / 查重 / 二次保存 | `system/phase1_protocol_driver.py` | 名称阶段前半段，最终拿到 `busiId` |
| 名称补充信息 | `system/phase2_protocol_driver.py` step 1 / 2 / 7 | 技术上放在 phase2，业务上仍属于名称阶段 |
| 投资人 / 股东摘要 | `system/phase2_protocol_driver.py` step 3 / 4 / 5 / 6 | 名称阶段后半段 |
| 名称提交 / 成功页 | `system/phase2_protocol_driver.py` step 8 / 9 | 完成 `nameId` 获取 |

## 2.5 第一阶段完成判定

满足以下条件，才算第一阶段真正完成：

- 已得到 `busiId`
- 已得到 `nameId`
- 已到达 `NameSuccess` 成功页
- 后续可以从“我的办件 / 继续办理”进入设立填报

## 3. 第二阶段：信息填报

## 3.1 入口

第一阶段结束后，第二阶段不是简单“继续当前页”，而是业务上进入：

1. 我的办件
2. 找到对应名称的办件
3. 点击继续办理
4. 进入 `establish` 主流程

协议上对应：

- `matters/operate [before]`
- `matters/operate [operate]`
- `establish/loadCurrentLocationInfo`

## 3.2 UI 页面顺序（4540 个独主线）

从你给的截图和当前打通链路看，第二阶段业务页面应整理为：

1. 基本信息
2. 成员架构
3. 成员信息
4. 补充信息
5. 税务信息填报
6. 材料补充
7. 营业执照领取
8. 云帮办流程模式选择
9. 预电子文档 / 信息确认
10. 云提交成功页（停点）

其中第 1-8 步与你截图完全一致；第 9-10 步是页面点击保存后的后置环节。

## 3.3 第二阶段协议映射（4540）

| 页面 / 环节 | 协议步骤 | 当前代码位置 | 说明 |
|------|------|------|------|
| 办件入口预处理 | 10-12 | `phase2_protocol_driver.py` | 进入 establish 业务域 |
| 云帮办入口探测 | 13 | `step13_ybb_select` | 可选 load，失败可不中断 |
| 基本信息 | 14-15 | `step14_basicinfo_load` / `step15_basicinfo_save` | 第二阶段第一个核心表单 |
| 成员架构 | 16 | `step16_memberpost_save` | 角色与岗位保存 |

| 补充信息 | 19 | `step19_complement_info_advance` | 非公党建等 |
| 税务信息填报 | 20 | `step20_tax_invoice_advance` | 税务填报推进 |

## 3.4 1151 有限公司差异

`1151` 在第二阶段会比 `4540` 多出若干页面，主要包括：

- `Rules`（章程）
- `MedicalInsured`
- `YjsRegPrePack`
- 更复杂的 `ComplementInfo` / `MemberPost` / `MemberInfo`

因此：

- `4540` 主线是 **25 步**
- `1151` 主线是 **28 步**

但从业务视角看，仍然属于同一个“第二阶段：信息填报”。

## 4. 现有代码与业务阶段的错位

当前仓库里，**业务阶段**与**技术驱动器命名**并没有完全对齐：

- `phase1_protocol_driver.py`
  - 负责名称阶段前半段
  - 核心目标是把名称预检推进到拿到 `busiId`

- `phase2_protocol_driver.py` step 1-9
  - 技术上在 phase2 驱动器里
  - 业务上其实仍属于“第一阶段：名称信息”

- `phase2_protocol_driver.py` step 10+
  - 才真正进入“第二阶段：信息填报”

这就是目前最容易让人误解的地方：

**代码里的 `phase2`，并不完全等于业务上的“第二阶段”。**

## 5. 开发框架应如何按流程编排

后续框架必须按下面的层级组织，而不是直接按“文件名叫 phase1/phase2”来理解。

## 5.1 正确的编排层级

### 第一层：业务阶段

- `阶段一：名称信息（核名）`
- `阶段二：信息填报`

### 第二层：页面子流程

阶段一页面子流程：

- 名称预检
- 名称补充
- 投资人/股东摘要
- 名称提交

阶段二页面子流程：

- 办件入口
- 基本信息
- 成员架构
- 成员信息
- 补充信息
- 税务信息
- 材料补充
- 营业执照领取
- 云帮办模式选择
- 预电子文档
- 云提交停点

### 第三层：协议步骤

- 由 `system/phase1_protocol_driver.py`
- 由 `system/phase2_protocol_driver.py`
- 统一通过 `get_steps_spec()` / step 函数对外提供

### 第四层：实现细节

- `phase2_bodies.py`：body 构造
- `phase2_enums.py`：字典映射
- `icpsp_crypto.py`：RSA / AES
- `icpsp_api_client.py`：唯一出网口

## 5.2 推荐的框架职责划分

| 层级 | 应承担的职责 | 当前落点 |
|------|------|------|
| Router 层 | 对外暴露 API，不解释业务细节 | `phase1_service/api/routers/` |
| Adapter / Core 层 | 以“阶段”和“页面”为单位编排 | `driver_adapter.py` / `phase2_adapter.py` |
| Driver 层 | 以“协议步骤”为单位执行 | `phase1_protocol_driver.py` / `phase2_protocol_driver.py` |
| Builder / Infra 层 | body、枚举、加密、上传、HTTP | `phase2_bodies.py` 等 |

## 5.3 后续应该新增的编排视图

建议在现有 `STEPS_SPEC` 之外，再显式增加两套上层视图：

### 业务阶段视图

- `BUSINESS_STAGE_SPEC`
  - `stage1_name_flow`
  - `stage2_establish_flow`

### 页面流程视图

- `PAGE_FLOW_SPEC_4540`
  - `basic_info`
  - `member_post`
  - `member_info`
  - `complement_info`
  - `tax_invoice`
  - `sl_upload_material`
  - `business_licence_way`
  - `ybb_select`
  - `pre_electronic_doc`
  - `pre_submit_success`

这样外层编排就能做到：

- 面向用户：按“两个阶段”表达
- 面向开发：按“页面”调试
- 面向协议：按“step”执行

## 5.4 当前最合理的落地方案

短期内不必推翻现有驱动器，只需要统一解释口径：

1. `phase1_protocol_driver` = 第一阶段前半
2. `phase2_protocol_driver step1-9` = 第一阶段后半
3. `phase2_protocol_driver step10+` = 第二阶段正式信息填报
4. API / CLI / Runner 对外都按“两阶段”暴露能力

---

## 6. 建议作为项目统一口径的最终结论

以后统一按下面这句话理解整个项目：

> **本项目业务上只有两个阶段：第一阶段是名称信息（核名），第二阶段是信息填报；协议层虽然拆成多个 step 和多个 driver，但所有代码编排都必须回到“业务阶段 → 页面流程 → 协议步骤”这三层映射上。**

这句话应作为：

- 文档口径
- API 设计口径
- CLI 命令口径
- Runner / Orchestrator 编排口径
- 断点续跑和日志归档口径

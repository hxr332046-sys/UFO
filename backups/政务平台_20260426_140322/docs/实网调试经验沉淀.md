## 2026-04-20 第一阶段（名称登记）阶段性沉淀

### 1) 旧草稿上下文污染新建流程
- 现象：明明带了 `entType=1100` 入口参数，页面仍回到旧“个人独资”名称流程。
- 证据：
  - 本地存储 `_topnet_flowParams` 含旧草稿上下文。
  - 页面标题/步骤和名称候选始终显示旧草稿（如“飞天有…”）。
- 根因：`flow-control` 初始化优先读取既有续办上下文。
- 处置：
  - 清理 `_topnet_flowParams` / `continueFlag`；
  - 调 `flow-control.initData()` 重建无 `busiId` 的新建态；
  - 验证 `flowData.busiId == null` 且 `entType == 1100`。
- 防回归：
  - 脚本启动前先做“上下文体检”，若发现旧 `busiId` 先清理再跑。

### 2) my-space“删除”按钮需要两段式确认
- 现象：点击“删除”后记录不消失，也无可见弹窗。
- 证据：
  - 抓包显示仅发起 `matters/operate`（`dealFlag=before`）；
  - 返回 `resultType=2`，提示“此操作将永久删除, 是否继续?”。
- 根因：只执行了 before 阶段，未走 after 确认提交。
- 处置：
  - 记录并尝试二段确认（`dealFlag=after`）；
  - 若前端确认框不稳定，保留接口证据并人工确认删除。
- 防回归：
  - 删除动作统一按 “before -> 校验提示 -> after” 三步执行并记录响应。

### 3) 名称核查页 `nameCheckRepeat` 间歇性 `A0002`
- 现象：`保存并下一步` 时失败，提示“服务端异常”。
- 证据：
  - `POST /register/name/component/NameCheckInfo/nameCheckRepeat`
  - 返回 `{"code":"A0002","msg":"服务端异常"}`
- 根因：服务端阶段性异常（非单纯前端点击问题）。
- 处置：
  - 执行“刷新 -> 复测 -> 重新抓包”；
  - 保留失败请求/响应与页面状态快照。
- 防回归：
  - 所有 A0002 自动触发一次刷新重试；
  - 将“是否发出请求”与“是否落办件”分开验收，避免误判。

### 4) 第一阶段验收口径固定
- 验收标准：**只看“名称登记完成后，‘我的办件’出现新‘填写中’记录”**。
- 不纳入第一阶段：设立登记、签章、云提交。
- 防回归：
  - 每次回报必须包含“我的办件列表快照”。

### 5) 手动成功链路（对照基线）
- 现象：用户手动可稳定完成“名称登记申报成功”，并在“我的办件”出现新记录。
- 证据：
  - 页面结果出现“申报成功/此名称可以申报”；
  - 办件列表新增对应“企业开办-名称已申请/填写中”记录；
  - 全局监听导出文件：`dashboard/data/records/global_listener_trace_latest.json`。
- 根因（对自动化差异）：
  - 自动化早期被旧草稿上下文污染；手动路径可通过真实页面交互完成正确上下文切换。
- 处置：
  - 先做上下文解绑（`flowParams/continueFlag`）；
  - 再按手动顺序进入名称检查并提交；
  - 全程监听接口与路由，作为回放基线。
- 防回归：
  - 每次异常先与“手动成功链路”对比：当前 hash、接口是否发出、是否出现申报成功页。

### 6) 2026-04-20 晚间复跑：自动链路被 4540 待办回拉
- 现象：本轮从 `entType=1100` 入口启动后，仍被回拉到“我的办件”，无法进入目标名称核查提交态。
- 证据：
  - `run_phase1_from_case_latest.json` 中 `guide_base_advance_phase1` 显示 `diverted_to_my_space`；
  - 同文件 `my_space_resume_continue` 仅命中 `entType=4540` 的“陈飞李（...个人独资）”记录，未发现 `entType=1100` 的续办路由；
  - “我的办件”快照文本仅见 `陈飞李（广西容县）食品市（个人独资）` 为“填写中”。
- 根因：平台当前可续办上下文与目标新建上下文不一致，且列表中存在更强绑定的非目标待办，导致自动恢复逻辑无可用目标路由。
- 处置：
  - 已执行“监听重装 -> 入口探测 -> continueFlag 清理 -> 第一阶段脚本复跑”完整流程并留档；
  - 将本轮归档为“反例样本”，用于优化“新建态识别优先级”。
- 防回归：
  - 新增硬性校验：进入续办前必须同时满足 `entType=1100` + `busiType=02_4`，否则不允许“继续办理”；
  - 增加“新建态直达失败即中止并提示人工一步”的分支，避免被 4540 待办污染。

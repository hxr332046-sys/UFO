## 四、铁律清单（刻在墙上）

### 铁律 1：两个 signInfo 魔数

```python
SIGN_INFO_NAME       = -252238669   # Phase 1 (name 域)
SIGN_INFO_ESTABLISH  = -1607173598  # Phase 2 establish 域
```

### 铁律 2：字段名大小写字符级匹配

- `cerno`（小写 n）✅ SlUploadMaterial 专用
- `cerNo`（大写 N）❌ → Jackson 丢 null → NPE → A0002

### 铁律 3：Phase 2 establish 阶段 busi_id 两种形态

- **初次创建**：`flowData.busiId=null`，BasicInfo save 成功后服务端分配新 busiId
- **断点续跑**：用已知的 establish busiId（即 Phase 1 的 busi_id 在 establish 域被复用）

### 铁律 4：load 响应的 `busiData` 不要整体回传

- load 返回的 `entDomicileDto` 有 **190+ 元数据字段**（estateType/houseCheck/legalDocDeliverAddress 等服务端内部字段）
- 回传会触发 A0002。**只保留 mitm 实录样板里的 41 字段**，其他用 case 覆盖

### 铁律 5：两次 save 才成功

- **第 1 次** BasicInfo save：`continueFlag=null` → `code=00000, resultType=2`（警告"请确认经营范围"）
- **第 2 次** save：`continueFlag="continueFlag"` → `code=00000, resultType=0`（成功，创建 busiId）
- 本项目默认一律传 `continueFlag="continueFlag"`，跳过警告确认

### 铁律 6：前端自动推进 — save 成功后不用 load 下一组件

- `save(compA)` 成功 → 服务端自动把 session 迁移到下一组件 `compB`
- 调用方连续调 `save(compB)` 即可，**不需要手动 load(compB)**

### 铁律 7：攻克过的别重跑

> 读 MD 文档 → 找到现有 busiId → 从断点续传 → 比从头打通省 10 倍时间

  -H "Content-Type: application/json" `
  -d $body

## 六、边界声明（本阶段明确不 cover）

| 项 | 原因 |
|---|------|
| 真正云提交（`status=90 → status=...` 最终签章提交） | 用户明说不点，且需要 CA U-key 硬件 |

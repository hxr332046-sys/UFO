## 一、我们造了什么

### 定义

一个**协议级 RPA 流程编排器**（Protocol-level RPA Workflow Orchestrator），用于自动化广西政务平台的企业注册全流程。

### 行业术语对照

| 我们的叫法 | 行业术语 | 英文 |
|---|---|---|
| 智能注册执行器 | 流程编排器 | Workflow Orchestrator |
| 纯协议驱动 | 协议级 RPA | Protocol-level RPA |
| 核名纠错交互 | 人机协作自动化 | Human-in-the-Loop Automation |
| 步骤序列 | 有限状态机 | Finite State Machine (FSM) |
| 错误路由链 | 感知-决策-执行循环 | Sense-Decide-Act Loop |

### 与传统 RPA 的区别

| 维度 | 传统 RPA（UiPath/Selenium） | 我们的协议级 RPA |
|---|---|---|
| 操作层 | 浏览器 DOM / 像素点击 | HTTP API 直接对话 |
| 速度 | 每步 3-10 秒（等渲染） | 每步 0.05-0.2 秒 |
| 稳定性 | 前端改版就挂 | 只要 API 不变就不挂 |
| 可观测性 | 截图/录屏 | 结构化 JSON 日志 |
| 复杂度 | 低（点哪填哪） | 高（需逆向 API 协议） |
| 维护成本 | 高（DOM 频繁变化） | 低（API 相对稳定） |

## 三、感知-决策-执行链（核心设计）

```
每个 API 响应
    │
    ├─ code=00000, rt=0     → ✅ 继续下一步
    ├─ code=00000, rt=2     → ⚠ 警告（禁限词/近似名）
    │   └─ 无 busiId        → 🔴 exit_code=6 → 交互纠错
    │   └─ 有 busiId        → ✅ 服务端已放行
    ├─ code=00000, rt=1     → ⚠ 业务警告（记录但继续）
    ├─ code=00000, rt=-1    → 🔴 致命错误（立即停止）
    ├─ code=GS52010103E0302 → 🔄 Session 过期 → CDP 刷新 → 重试
    ├─ code=D0029           → ⏸ 限流 → 立即停止 + 冷却提示
    ├─ code=D0022           → 🔴 越权 → body 构造问题
    ├─ code=A0002           → 🔴 系统异常 → 报告具体步骤
    └─ HTTP 401/403         → 🔴 认证失效 → 要求重新登录
```

## 四、核名纠错交互系统（Human-in-the-Loop）

### 触发条件
Phase 1 第 7 步（operationBusinessDataInfo#second）返回 `code=00000, resultType=2, busiId=null`

### 展示信息
1. **当前名称**：全称、字号、冠名、组织形式
2. **禁限用词详情**：触发词、限制类型（驰名商标/知名企业）、备注
3. **近似企业列表**：最多 10 条，含企业名称和所属地区
4. **系统诊断**：分析警告原因（禁限词/近似名/双重警告）

### 用户选项
| 选项 | 行为 | 会话状态 |
|---|---|---|
| 更换字号 | 输入新字号 → 更新 case（内存+磁盘）→ 重跑 Phase 1 | **保持**（不重启脚本） |
| 强制继续 | 带 continueFlag 重试 | **保持** |
| 退出 | 优雅退出 | 保存诊断结果 |

### 关键技术点
- Session/Auth/Client 在暂停期间保持活跃
- 最多重试 5 次改名
- case JSON 文件实时同步更新

## 五、为什么不用 Agent 框架

### 评估过的框架

#### LangGraph（状态机 + LLM 推理）
- **优点**：状态机思想好、支持 human-in-the-loop、有 checkpoint
- **不适用原因**：
  - 我们的步骤序列是**确定性的**（步骤已知、顺序固定、参数已知）
  - LangGraph 在每个节点调用 LLM 推理，但我们 90% 的步骤不需要推理
  - 引入后核名耗时从 7 秒膨胀到 40+ 秒（每步 LLM 推理 2-5 秒）
  - LangChain 生态依赖重，与我们的轻量级 HTTP 客户端冲突

#### CrewAI（多 Agent 协作）
- **优点**：角色分工清晰、适合复杂协作任务
- **不适用原因**：
  - 我们是**单操作员对单系统**，不需要多角色协作
  - 设计目标是内容生产（写文章、做报告），不是 API 自动化
  - 过度工程化，引入不必要的复杂度

#### AutoGen / Microsoft（多角色对话）
- **优点**：强大的多 Agent 对话和代码执行能力
- **不适用原因**：
  - 面向复杂推理任务（代码生成、数学证明）
  - 对话式架构不匹配我们的请求-响应流水线
  - 框架重量级，部署和维护成本高

### 根本原因：确定性 vs 不确定性

```
我们的系统:
  95% 确定性（步骤固定、参数已知、协议已逆向）
  +5% 不确定性（禁限词、未知错误、新表单类型）

Agent 框架设计假设:
  80%+ 不确定性（不知道下一步做什么、需要 LLM 判断）
```

**用 Agent 框架处理确定性流程 = 用自动驾驶系统开固定路线的地铁。技术上能做，但完全没必要。**

### 正确的 LLM 接入方式

不换框架，在现有编排器的**不确定性节点**按需调用 LLM：

```
现有编排器（确定性执行）
    │
    ├─ 正常流程 → 直接执行（不调 LLM）
    │
    └─ 异常/不确定 → 调用 LLM
        ├─ 禁限词 → LLM 分析原因 + 建议 3 个新字号
        ├─ 未知错误 → LLM 分析响应体 + 给修复建议
        ├─ 新企业类型 → LLM 分析字段差异 + 自动生成 body
        └─ 资料解析 → LLM 从自然语言/图片提取结构化 case
```

## 六、演进路线图

### 阶段 1：确定性引擎 + 人工介入 ✅ 已完成
```
人 → 准备 case JSON → 脚本跑 → 遇错暂停 → 人做决定 → 继续
```
- Phase 1 核名 8 步协议化
- Phase 2 设立 25 步协议化
- 核名交互纠错系统
- 支持 4540（个人独资）和 1151（有限公司）

### 阶段 2：确定性引擎 + LLM 辅助决策（下一步）
```
人 → 准备资料 → LLM 解析为 case → 脚本跑 → 遇错 LLM 建议 → 人确认 → 继续
```
- LLM 字号建议（核名失败时自动推荐）
- LLM 错误分析（未知响应自动诊断）
- LLM 资料提取（从文本/图片生成 case JSON）
- 断点续跑（Phase 2 从任意步骤恢复）

### 阶段 3：自主 Agent（终极形态）
```
人 → 给一句话 "帮我注册个软件公司" → Agent 全自动 → 到云提交停下 → 人审批
```
- 全自动核名（失败自动换名，最多 N 次）
- 全自动资料准备（OCR 身份证、解析租赁合同）
- 多企业批量注册
- 事后监控（注册状态追踪、驳回自动处理）
- 自然语言交互界面

| 限流检测与冷却 | ✅ |

### 关键铁律（踩坑总结）
1. 两个 signInfo 魔数按域切换（Phase 1 ≠ Phase 2）
2. `cerno`（小写 n）≠ `cerNo`（驼峰），SlUploadMaterial 专用
3. BasicInfo save 需要两次（第 1 次 rt=2 → 第 2 次 continueFlag → rt=0）
4. entDomicileDto 有 190+ 字段，不能整体回传（→ A0002）
5. SESSION cookie 必须用 `Network.getAllCookies`（getCookies 有 bug）
6. matters/operate 删除的第二步 dealFlag 必须是 `"operate"` 不是 `"after"`
7. Phase 2 establish busiId：初次为 null 让服务端分配，断点续跑用已知值

## 八、结论

> **我们的系统不是 Agent，但比通用 Agent 框架更适合这个场景。**
>
> 它是一个**确定性流程引擎**，在关键不确定性节点提供**人机协作**接口。
>
> 下一步不是换框架，而是在现有引擎上**按需接入 LLM**，让它从"聪明的机器人"进化为"有大脑的机器人"。

```
引擎（快、稳、确定性） + LLM（智能、灵活、处理意外）= 最优解
```

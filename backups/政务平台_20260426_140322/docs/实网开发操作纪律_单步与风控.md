# 实网开发操作纪律：单步、延时、防风控

> **适用**：广西 9087 `icpsp-web-pc` 真实环境下调试、填表、跑通至**云提交文案停点**（不点最终提交）。  
> **与案例资料绑定**：`docs/资料.txt`（广西容县李陈梦等测试数据；材料路径以本机为准）。  
> **心态**：慢慢来、不赶进度；**一次只推进一层因果**，避免触发长时间风控。

---

## 1. 唯一允许的推进节奏（必须记住）

```
提交（一步副作用） → 类人延时 → 读反馈（DOM / 接口 / 提示 / 快照） → 再决定下一步
```

- **提交**：一次导航、一次点击、一次 `eval`、**一个** file 槽位上传等，都算「一步」。  
- **延时**：统一走 `sleep_human` + `config/human_pacing.json`（默认 **`min_delay_sec` ≥ 1.0**），见 `system/human_pacing.py`。  
- **分析反馈**：看清报错文案、级联是否完成、是否出现新弹窗，再编码下一步。  
- **下一步**：在**确认上一步结果**之后再执行；**不得**假设页面已就绪就连点。

这与仓库 **[LLM政务自动化框架_目标与实施标准.md](./LLM政务自动化框架_目标与实施标准.md)** 中的「类人节奏」「顺序与节拍」一致；思维顺序见 **[逆向复现方法论_上下文先行与数据填充.md](./逆向复现方法论_上下文先行与数据填充.md)**。

---

## 2. 明确禁止（碰了容易风控几小时）

| 禁止项 | 说明 |
|--------|------|
| **死循环 / 循环脚本** | 不用 `while True` 对同一页面**连续**点主按钮、连发导航、连打接口试密码式探测。轮询仅允许**有上限、有退避**的等待（如 `wait_href` 等），且间隔用短 `time.sleep`，不冒充「下一步业务决策」。 |
| **并发提交** | 不开多进程/多线程/多标签同时对 9087 做「带副作用」的操作；业务上**有先后依赖**（先登录 → 再专区 → 再表单 → 再上传），必须**串行**。 |
| **毫秒级连点** | 不等渲染、不等请求返回就叠下一步；等同于对站点施压，易触发风控。 |
| **实网压速** | 对生产站点**不要**长期用 `--human-fast` 当常态；调试可短暂使用，见 `.cursor/rules/gov-cdp-pacing.mdc`。 |

## 3. 与「资料」相关的目标停点（本仓库已约定）

- **业务分两阶段**（核名 → 办件中心接续办理 → 长表单至云提交）：详见 **[设立登记两阶段流程.md](./设立登记两阶段流程.md)**。自动化须区分「名称登记结果」与「我的办件 → 继续办理」后的**第二阶段**，不可混为一条无状态链。  
- **目标**：按 `资料.txt` 推进流程，收集**错误类型、提示、接口与 UI 反馈**，直至 **云提交相关文案出现即停**，**不执行云提交、不进入云处理**（与案例脚本、验收线一致）。  
- **材料路径**：身份证、租赁合同等为**本机绝对路径**；换机器需改 `config/rehearsal_assets_*.json` 或案例配置，**不要**在循环里自动扫盘重试。

---

## 4. 工程侧检查清单（开发前扫一眼）

- [ ] 脚本入口已 `configure_human_pacing(...)`。  
- [ ] 每个 CDP 副作用后接 `sleep_human`（上传每槽位后已有间隔的保持）。  
- [ ] 主路径逻辑是**有限状态 / 有限步数**，出错则**记录并退出**，而不是无限重试同一动作。  
- [ ] 同一时间只跑**一个**实网自动化进程。

## 一、方案原理

### 核心思路：一次普查 → 永久复用

| 阶段 | 操作 | 频率 |
|------|------|------|
| **普查** | 遍历所有桌面程序控件树，提取静态信息 | 一次性（安装新软件/更新UI时重做） |
| **存档** | 保存 automation_id、control_type、class_name、static name | 持久化 JSON |
| **直达** | 运行时按存档索引直接定位控件，跳过整棵树遍历 | 每次操作 |

### 静态 vs 动态信息

| 可存档（静态） | 不可存档（动态） |
|----------------|-----------------|
| Automation ID | HWND 窗口句柄 |
| Control Type | 控件位置/矩形 |
| Class Name | 列表内容 |
| 控件树结构 | 输入框文本值 |
| 窗口标题模式 | 控件 enabled/visible 状态 |

## 五、使用方法

### 运行时直达操控（3步）

```python
from desktop_quick_control import find_app_window, connect_uia, control_by_auto_id, control_by_name

# Step 1: 找到目标窗口（按进程名+标题）
info = find_app_window("chrome.exe")

# Step 2: UIA连接
app = connect_uia(info["pid"])
top_win = app.top_window()

# Step 3: 直达控件（按存档索引）
addr_bar = control_by_auto_id(top_win, "view_1012")  # 地址栏
addr_bar.set_edit_text("https://www.google.com")     # 输入URL
```

### 重新普查（安装新软件后）

```bash
python desktop_control_census.py
# → 更新 desktop_control_archive.json
```

## 六、局限性

| 限制 | 说明 |
|------|------|
| **微信控件不可达** | Qt框架，UIA仅暴露 TitleBar，内部控件无法枚举 |
| **哔哩哔哩控件不可达** | Chromium内核但UIA无法枚举内部 |
| **Chromium内部DOM** | 仅能操控浏览器外壳（工具栏/地址栏），网页内容需 DevTools/Puppeteer |
| **Automation ID可能变化** | Chrome大版本更新后 view_xxx 编号可能重排 |
| **存档需定期刷新** | 新安装软件或UI更新后需重新普查 |

### 发送消息流程（双屏安全方案）

```
1. set_focus()        → 激活 Windsurf 窗口（不受屏幕位置限制）
2. Ctrl+L             → 键盘快捷键聚焦 Cascade 输入框
3. 剪贴板 Ctrl+V      → 粘贴中文消息（唯一可靠的中文输入方式）
4. Enter              → 发送消息
```

**关键代码**：
```python
from pywinauto import Application, keyboard
import pyperclip

app = Application(backend="uia").connect(process=pid, timeout=5)
top_win = app.top_window()
top_win.set_focus()
time.sleep(1)

keyboard.send_keys('^l')       # Ctrl+L 聚焦输入框
time.sleep(1.5)

pyperclip.copy(message)       # 复制到剪贴板
keyboard.send_keys('^a')      # 全选
keyboard.send_keys('^v')      # 粘贴
time.sleep(0.5)

keyboard.send_keys('{ENTER}') # 发送
```

### Cascade 控件结构

| 层级 | class 特征 | 说明 |
|------|-----------|------|
| 根 | `chat-client-root windsurf-theme-target` | Cascade 面板根 |
| 滚动区 | `cascade-scrollbar flex w-full` | 所有消息的父容器 |
| 消息气泡 | `panel-border panel-bg text-ide-text-color` | 单条消息容器 |
| 用户消息 | 短文本(1-2 fragments) | 通常含 Image 上传标记 |
| AI回复 | 长文本(40-99 fragments) | 含代码块、Markdown |
| AI命令 | `codeium-text-medium truncate` | "Command bun in …" |
| 思考状态 | `flex min-w-0 ... enter-` | "Thought for Xs" |
| 文件变更 | `group flex flex-row ... hover:bg-neutr` | "engine.ts + 72" |
| 终端 | `terminal xterm` + `xterm-accessibility-tree` | 终端输出内容 |
| 输入框 | `min-h-[2rem] outline-none` (Edit) | "Ask anything (Ctrl+L)" |

### 双屏注意事项

- 第二显示器窗口坐标为**负值**（如 `rect=(-9,-1089)`），这是正常的虚拟屏幕布局
- `pyautogui.click()` 在负坐标下可能失败，**不要依赖坐标点击**
- 使用 `Ctrl+L` + 剪贴板方案**完全不受屏幕位置影响**
